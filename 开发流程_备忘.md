## 硬性规则
- 当 Codex 当前上下文是 `staging`、`feature/*`、测试版或新功能开发时，默认只能使用测试版 Render 和测试版 Supabase。
- 本地、`staging`、测试版和新功能分支内，Codex 默认自主完成完整开发闭环，不做许可式反复提问；包括但不限于读代码、改代码、跑测试、启动本地服务、调试、提交和推送测试分支、触发和验证 Render Staging 自动部署、验证测试版页面或 API、写入并清理明确的测试版临时数据。
- 测试版开发中遇到可合理判断的技术选择，Codex 应直接按最小可验证方案执行并在结果中说明取舍；不要把普通 staging 动作逐项提交给用户批准。
- 本地和测试版普通开发目标是快速完成、快速验证、把结果汇报清楚；用户不需要为每一步授权。
- 只有涉及 `main`、Production、正式 Render、正式 Supabase、生产数据、生产环境变量、正式部署、生产回滚、不可逆删除、付费外部操作，或环境映射不清且即将产生数据/部署影响时，Codex 才必须先停下来确认。
- Staging 上下文默认目标：Render `ltm-web-staging` / `https://ltm-web-staging.onrender.com` / Supabase `LTM WEB STAGING`。
- Main/Production 上下文默认目标：Render `ltm-web` / `https://ltm-web-gt13.onrender.com` / Supabase `LTM WEB`。
- 日常开发和测试不得直接使用正式版 Render URL。
- 日常开发和测试不得连接正式版 Supabase 数据库。
- 测试版只能连接 `LTM WEB STAGING`。
- 正式版只能连接 `LTM WEB`。
- 如果本地 `.env`、Render 环境变量或任务目标无法判断环境，Codex 必须先停下来确认，不得猜测。
- Render 套餐 plan 以 Dashboard 为准，`render.yaml` 不管理 `plan` 字段，避免 Blueprint 同步把 Standard 服务降回 Free。
- 任何 DB migration 或生产数据结构变更前，必须先按 `docs/backup_restore.md` 对目标环境做备份，并先在 staging 数据库验证恢复/迁移。
- 任何变更进入正式版之前，必须先在测试版验证通过，并得到用户明确确认。
- 不在代码仓库、文档或聊天输出中记录数据库密码、完整 `DATABASE_URL`、Deploy Hook 或其他密钥。

## 推荐开发流程
1. 在 `staging` 或 `feature/*` 分支开发。
2. 推送到 GitHub 后，由 Render 测试服务自动部署到 `https://ltm-web-staging.onrender.com`。
3. 在测试版完成浏览器可见验证和必要 API 验证。
4. 用户确认可以发布后，再把 `staging` 合并到 `main`。
5. Render 正式服务自动部署到 `https://ltm-web-gt13.onrender.com`。
6. 发布后检查正式版 `/api/health` 和核心页面。

## 优化类需求的需求解读与边界规则
- 对“性能优化、成本优化、egress 优化、生产化、安全加固、架构治理”这类横向需求，Codex 必须先把用户的业务前提列为第一优先级；技术指标只能在不破坏核心业务判断、主要操作流程和用户心智的范围内优化。
- 不能把“减少请求、减少数据量、分页、默认时间范围、摘要化、缓存、懒加载”等通用工程手段直接套到所有页面。每个改动必须先判断该页面是“核心业务判断入口”还是“明细查询/管理入口”。
- 核心业务判断入口默认保持原有业务语义：用户依赖全量、多年、多维、自动计算或连续刷新才能完成判断时，不得为了技术优化减少数据范围、改变默认筛选、取消自动行为或改成摘要结果，除非用户明确确认。
- 明细查询/管理入口可以优先考虑分页、字段白名单、加载更多、短 TTL 缓存和减少重复请求，但必须保证页面上有清楚的继续加载/分页反馈，不能让用户误以为数据丢失。
- 遇到会改变用户可感知行为的优化，必须在实施前列出“用户会看到什么变化”。包括但不限于：默认显示数据变少、需要点击加载更多、自动刷新变为手动、全量数据变为摘要、筛选默认值变化、按钮可见性变化、耗时反馈变化。
- 前端显示状态必须和后端查询语义一致。页面显示“全选”“全部”“实时更新”“刷新”“加载更多”“访客可看”等状态时，后端实际查询和权限结果必须与该状态一致；不能只验证 API 200 或页面无报错。
- 当技术优化和业务体验存在取舍时，Codex 不能替用户做产品判断。必须停下来确认，例如：是否允许默认只看最近时间范围、是否允许某个页面分页、是否允许自动计算降频、是否允许摘要替代明细。
- 优化类任务的验收不能只看测试通过和接口指标，必须补充真实用户路径验证：从页面入口进入、按用户常用筛选操作、观察展示结果、确认按钮行为、确认数据范围、确认刷新/计算效果。
- 后续复盘不只记录具体功能点错误，还要记录需求理解偏差：是否把技术目标置于业务前提之上，是否误用了通用工程最佳实践，是否没有识别核心业务路径，是否在不确定边界时没有向用户确认。

## Codex 测试连通性规则
- Codex 终端命令访问 Render 偶发出现 `SSL connection timeout`、外网权限审批失败、`ERR_BLOCKED_BY_CLIENT` 或没有 HTTP 状态码时，优先判定为本机/工具测试通道问题，不直接判定测试版网站故障。
- 每次开始新的浏览器验证前，Codex 必须先关闭之前打开过的本项目测试页签（包括 `ltm-web-staging.onrender.com`、本地测试页或带 `?codex=` 的验证页），再新开干净页签测试；避免旧页签缓存、旧登录态或过多标签页影响判断。
- Staging 页面验证优先使用 Codex 内置浏览器打开 `https://ltm-web-staging.onrender.com/?codex=<commit>`，检查 URL、页面标题、控制台错误、静态资源版本和目标功能可见状态。
- `python3 scripts/check_staging_health.py` 和 `curl` 只作为辅助连通性检查；只要内置浏览器能打开测试版并确认资源版本/页面状态，就可以判定测试版页面已部署生效。
- 只有拿到明确 HTTP 4xx/5xx、页面功能错误、API 业务错误，或用户浏览器也复现异常时，才进入 Render/app/Supabase 故障排查。
- 浏览器可见验收优先以 Codex 内置浏览器、用户实际 Chrome 或可见页面为准；命令行探测只作为辅助信号。


## 版本记录
- 本项目使用 `版本更新记录.md` 记录测试版更新和正式版发布。
- Staging 更新记录：记录变更内容、验证内容、是否准备进入正式发布。
- Production 发布记录：记录发布日期、发布内容、来源分支、main commit / Render deploy（可获取时）、数据库影响、正式验证结果、回滚点。
- 记录中不得写入密码、完整 `DATABASE_URL`、Deploy Hook、API Key 或其他密钥。

## 本地端口说明
- `8001`：本地开发端口，适合日常调试。
- `8000`：本地近似正式运行端口，只用于本机冒烟验证。
- 本地 `8000` 不再等同于真正正式版；真正正式版以 Render 正式 URL 为准。
- 本地运行时必须确认 `.env` 指向测试数据库，除非用户明确要求操作正式数据。

## 回滚
- 代码层面优先通过 Git 回滚或重新部署上一个已确认 commit。
- 如使用 `releases/vX.Y_日期/` 备份包回滚，必须先确认目标环境是测试版还是正式版。
- 正式版回滚前必须得到用户明确确认。

## 原则
- 最小改动，不动已有功能代码。
- `sed` 必须用正则匹配文本，绝不用行号。
- 改完先在测试环境验证，再考虑正式发布。
- 改 HTML/JS 后提醒 Cmd+Shift+R 强制刷新。
