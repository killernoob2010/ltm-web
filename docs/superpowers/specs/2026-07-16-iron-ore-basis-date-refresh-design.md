# 期现页面最新数据日期刷新与去重设计

## 目标

修复期现页面长时间打开后，左上角“最新数据日期”仍停留在旧值的问题，并删除最优仓单区域重复的“数据截至”日期。

## 根因

`frontend/iron_ore_basis.js` 仅在管理页或展示页首次初始化时请求 filters 接口。`managementInitialized` 和 `displayInitialized` 置为 `true` 后，后续重新进入页面不再刷新顶部日期，因此数据库已从 2026-07-14 更新到 2026-07-15 时，已打开的页面仍可能显示旧值。最优仓单则每次重新请求，导致两处日期可能既重复又不一致。

## 实现设计

1. 保留现有 filters 接口和首次筛选器初始化逻辑。
2. 将“请求 filters 并更新最新日期”与“初始化复选器及绑定事件”分开。
3. 每次进入期现数据管理或期现数据展示时，都请求对应 filters 接口并更新左上角日期。
4. 只在首次进入时构建复选器和绑定事件，不重置用户已选的年份、品种、港口或分页状态。
5. 最优仓单仍每次进入时重新读取数据，但不再显示“数据截至”文案。
6. 更新期现 JavaScript 静态资源版本，确保 Render Staging 浏览器加载新代码。

## 验收标准

- 当 filters 接口的 `latest_data_date` 从 2026-07-14 变为 2026-07-15 时，不刷新整个网页，离开再进入期现页面后，左上角必须显示 2026-07-15。
- 期现数据管理和期现数据展示都执行日期刷新。
- 重新进入页面不会重复绑定复选器事件，不会重置已选筛选条件。
- 期现数据展示页只有左上角一处数据日期，最优仓单标题区不再显示“数据截至”。
- Render Staging 真实页面日期与 `iron_ore_basis_results` 的最大 `business_date` 一致。

## 验证计划

- 新增前端回归测试，覆盖每次激活都刷新日期、筛选器只初始化一次、重复日期文案已删除。
- 运行期现前端专项测试和 Node 全量回归。
- 推送 `staging` 后，在真实测试页执行“进入期现页面 -> 切换到其他页面 -> 再进入期现页面”，核对日期、唯一性、筛选状态和控制台。

## 范围与边界

- 评估：D1 / T1 / R1 / C1。
- 仅修改期现前端日期刷新和展示。
- 不修改数据库、filters 接口返回结构、自动同步调度、EBC/新浪取数、基差计算或已有数据。
- 仅发布和验证 Staging；不操作 `main`、Production Render 或正式数据库。
