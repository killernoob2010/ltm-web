# 铁矿石期现基差 API 自动增量更新设计

**日期：** 2026-07-14

**状态：** 业务设计已确认，等待书面规格复核

**目标环境：** 先在 Render Staging 人工触发验证；Production 发布、环境变量和数据库变更另行确认

## 1. 目标与最终效果

在现有轻量化交易管理系统中延续同一条铁矿石期现基差时间序列：

- 保留已经由 Excel 初始化的历史结果，日期范围为 2024-01-02 至 2026-07-10；
- 从历史截止日后的第一个交易日 2026-07-13 开始，通过 EBC 日频现货数据和新浪 `I0` 主力连续日线自动计算增量基差；
- 此后每个已完成交易日自动新增数据；
- 现有“期现数据管理”和“期现数据展示”继续读取同一套结果表，页面不区分 Excel 历史段与 API 增量段；
- 两个现有页面顶部增加一条“最新数据日期”提示，让用户直接确认数据已经更新到哪一个交易日；
- 自动更新运行在现有 Render Standard Web Service 中，不依赖用户的 Mac 开机、浏览器登录或打开网站；
- 不新增网站、独立程序、Render Cron Job 或新的 Render 服务。

最终业务时间线：

```text
2024-01-02 ───────── 2026-07-10 | 2026-07-13 ───────── 以后每个交易日
       已验收 Excel 历史结果       |       API 自动计算结果
                                  ↑ 同一结果表、同一查询接口、同一页面
```

## 2. 已确认的现状

### 2.1 系统与运行环境

- Production `ltm-web` 和 Staging `ltm-web-staging` 当前均运行在 Render Python 3 Standard 服务上；
- 后端由 `backend/app/main.py` 启动，现有部分缓存和后台更新任务已随 Web Service 常驻运行；
- 复用现有 Standard 服务不会新增 Render 服务或 Cron Job 费用；
- Staging 默认不启用自动定时任务，避免测试版和正式版同时登录同一个 EBC 账号。

### 2.2 现有期现模块

- `iron_ore_basis_results` 保存页面使用的精简期现结果；
- `iron_ore_basis_details` 保存一一对应的完整计算明细；
- 当前共有 60,424 条结果和 60,424 条明细，覆盖 15 个品种、8 个港口；
- 现有前端通过 `backend/app/iron_ore_basis.py` 的只读 API 查询，不需要因数据来源变化而重做页面；
- `backend/app/iron_ore_basis_import.py` 负责校验并导入 Excel 中已经计算好的结果，不包含可供 API 更新复用的基差计算引擎。

### 2.3 数据源可用性验证

- 新浪 `I0` 历史日线已从 Render Staging 成功抓取；
- EBC 登录接口和日频指标查询接口已从 Render Staging 成功调用；
- EBC 目标指标为日频序列，一个指标在一个日期对应一个值，不是秒级或分钟级电子盘口；
- 2026-07-14 早间实测目标 EBC 日频指标最新结束日期为 2026-07-13；
- EBC 登录取得的访问令牌只用于当次任务，不需要用户 Mac 或浏览器保持登录。

以上验证只证明当前接口路径可用，不构成第三方接口永久稳定性承诺。同步任务必须保留失败记录、重试和可关闭能力。

## 3. 已确认的业务规则

### 3.1 正式数据日期

- 不使用正在变化的交易日内 `I0` 数据生成正式基差；
- 每次任务明确指定一个已经完成的目标交易日；
- 只有目标日期的 EBC 日频价格与 `I0` 收盘价同时存在时，才生成对应期现结果；
- 周末和节假日不虚构日期，也不把前一交易日价格平移到非交易日；
- 初次启用时从数据库最新结果日的下一个真实交易日起逐日补齐，当前切点为 2026-07-13。

### 3.2 调度时间

时区固定为 `Asia/Shanghai`：

| 运行时间 | 目标 | 作用 |
|---|---|---|
| T 日 21:30 | T 日 | 日盘收盘后的第一次更新 |
| T+1 日 09:30 | T 日 | 第一次补检 |
| T+1 日 10:30 | T 日 | 第二次补检 |

21:30 时下一交易日夜盘可能已经开始，但任务仍显式查询已经结束的 T 日，不使用夜盘正在变化的新交易日数据。

Render 服务启动或重启后还会执行一次缺口检查：从结果表最新日期之后开始，补到两个数据源共同具备数据的最近已完成交易日。日常检查至少回看最近 10 个自然日，用于弥补服务重启、节假日和临时接口失败造成的缺口。

### 3.3 历史值与去重

- EBC 和 `I0` 的 T-1 及更早数据在本系统中视为正式来源值；
- 来源记录不存在时新增；
- 同一来源、指标、日期已存在且数值相同，直接跳过；
- 已存在的历史来源值与接口新返回值不同时，不自动覆盖，只记录新旧值和异常原因；
- Excel 历史结果不重新导入、不回填到新原始数据表、不因 API 上线而重算；
- API 原始数据表从自动更新启用日开始积累。

### 3.4 缺失组合与最新日期

- 每个日期、港口、品种组合独立判断完整性；
- EBC 价格、`I0` 收盘价和当前规则参数齐全的组合立即写入；
- 缺少 EBC 数据的组合先跳过，不阻塞同日其他完整组合；
- 21:30、次日 09:30、10:30 共三次尝试；
- 第三次仍缺失的组合继续跳过，后续启动补检和日常回看仍可自动补齐；
- 初版不识别停更、不生成异常状态、不展示缺失数量；
- 页面顶部只显示结果表中的最新业务日期，例如“最新数据日期：2026-07-14”；
- 如果一个目标日期没有任何完整组合成功写入，页面日期继续停留在上一个已有数据的交易日。

### 3.5 规则版本

- 当前已经确认的整套业务规则持续有效，不因跨年自动切换；
- “业务规则”包括品种参数、EBC 指标映射、高品位选择、特殊品种映射、干湿吨换算、五项质量调整、品牌升贴水、参数版本和规则版本；
- 只有用户明确提供并启用新规则时才创建新版本；
- 新规则从启用后的下一个真实交易日起生效；
- 新规则不回算、不覆盖启用日及以前的历史结果；
- 每条结果保存实际使用的规则版本和参数版本，保证时间序列可解释。

## 4. 方案选择

已比较的三种方式：

1. **Mac 本地定时抓取：** 实现简单，但依赖 Mac 开机联网，不满足业务目标；
2. **新增 Render Cron Job 或独立服务：** 隔离性好，但增加服务、费用和部署维护；
3. **复用现有 Render Standard Web Service 的后台调度：** 与项目已有后台任务模式一致，不新增服务和费用，部署链路最短。

采用方案 3。通过数据库级运行锁和幂等业务键处理服务重启或重复触发，不依赖单个进程“只运行一次”的假设。

## 5. 总体架构

```text
现有 Render Standard Web Service
│
├── FastAPI 现有网页和只读 API
│
└── 新增期现自动同步调度器
    ├── EBC 数据源适配器：登录、取得临时令牌、查询日频价格
    ├── 新浪数据源适配器：查询 I0 主力连续日线
    ├── 基差计算引擎：复现已确认 Excel 业务规则
    ├── 同步编排：目标日期、完整性、去重、事务和运行锁
    └── PostgreSQL / Supabase
        ├── 新增原始来源点表
        ├── 现有期现结果表
        ├── 现有期现计算明细表
        └── 新增同步运行日志表
```

页面调用关系保持不变：

```text
浏览器页面 → 现有 FastAPI 只读接口 → 现有期现结果表
```

页面不会在用户访问时实时调用 EBC 或新浪。第三方接口失败不会拖慢或阻断页面读取，页面始终只展示数据库中已经成功写入的完整组合。

页面仅额外读取系统自身的最新数据日期接口，不直接读取第三方数据：

```text
期现数据管理 / 期现数据展示 → 最新数据日期 API → 现有期现结果表
```

## 6. 后端组件边界

### 6.1 数据源适配器

建议新增 `backend/app/iron_ore_basis_sources.py`：

- `EbcBasisSource`：登录 EBC、查询已配置指标、标准化日期和数值；
- `SinaI0Source`：复用现有新浪历史行情调用方式，返回按日期索引的 `I0` 收盘价；
- 只负责获取与标准化，不计算基差、不写数据库；
- 返回结构化错误，不在异常、日志或返回值中输出账号、密码和访问令牌。

### 6.2 版本化规则包

建议新增 `backend/app/iron_ore_basis_rules.py`：

- 保存当前已经验收的品种参数、EBC 指标映射、高品位选择、特殊品种映射、质量调整参数、品牌升贴水、规则版本和参数版本；
- 当前规则包不设置自然年度终止日，持续有效到用户明确提供并启用新规则；
- 新规则以新版本加入，并设置为用户启用日后的下一个真实交易日起生效；
- 不提供网页编辑入口，规则变化仍按代码变更、测试版验证和生产确认流程发布。

### 6.3 计算引擎

建议新增 `backend/app/iron_ore_basis_calculation.py`：

- 把已验收工作簿中的计算规则固化为可测试的纯函数；
- 输入为一个日期的标准化 EBC 价格、`I0` 收盘价和当前有效规则包；
- 输出为现有结果表和明细表所需的完整行；
- 不访问网络、不决定调度时间、不直接提交数据库事务。

核心公式保持为：

```text
干吨现货价 = 湿吨现货价 / (1 - H2O)

标准化现货价 = 干吨现货价 - 质量升贴水 - 品牌升贴水

基差 = 标准化现货价 - I0 主力连续收盘价
```

计算引擎必须覆盖现有 15 个品种、8 个港口、EBC 高品位指标选择、昆巴粉和杨迪粉等特殊映射、Fe/SiO2/Al2O3/P/S 五项调整、品牌升贴水以及现行规则和参数版本。

### 6.4 同步编排与调度

建议新增 `backend/app/iron_ore_basis_sync.py`：

- 判断触发时间对应的目标交易日和缺口范围；
- 获取数据库运行锁，防止重复任务并发；
- 调用两个数据源适配器；
- 校验目标日期和预期指标完整性；
- 比较原始来源点并执行去重或历史变化告警；
- 调用计算引擎；
- 在单一事务中写入本轮完整组合的原始来源点、结果、明细和运行日志；
- 按 21:30、09:30、10:30 调度，并在进程启动时补检。

### 6.5 人工触发入口

建议新增 `scripts/sync_iron_ore_basis.py`：

- 调用与生产调度器完全相同的同步编排，不复制计算或写库逻辑；
- 默认只读预演，输出目标日期、来源完整性、预计新增、重复和缺失组合数量；
- 只有显式增加 `--apply` 才写入当前环境数据库；
- Staging 验收通过该入口人工触发，Production 定时任务不依赖该脚本进程常驻。

### 6.6 应用入口和现有 API

- `backend/app/main.py` 只增加调度器启动调用；
- `backend/app/db.py` 增加两张辅助表的兼容迁移；
- `backend/app/iron_ore_basis.py` 的现有结果查询契约不变，只新增或复用一个只读最新数据日期字段；
- 现有期现管理和展示页面顶部增加复用的单行“最新数据日期”组件；
- Excel 导入脚本继续保留，作为历史初始化和经过人工确认的应急补录入口。

## 7. 数据模型

### 7.1 原始来源点表

新增 `iron_ore_basis_source_points`，建议字段：

- `id`；
- `source_name`：`ebc` 或 `sina`；
- `indicator_code`；
- `indicator_label`；
- `business_date`；
- `value`；
- `unit`；
- `frequency`：固定记录为日频；
- `payload_sha256`：标准化来源点摘要，不保存完整敏感响应；
- `first_fetched_at`；
- `created_at`。

唯一键：

```text
source_name + indicator_code + business_date
```

该表只保存 API 上线后的计算依据，不回填过去三年的 Excel 历史来源点。

### 7.2 现有结果表和明细表

继续使用：

- `iron_ore_basis_results`；
- `iron_ore_basis_details`。

现有业务唯一键保持不变：

```text
日期 + 港口 + 品种 + 规则版本 + 参数版本
```

API 行仍保存完整计算明细。为减少迁移范围，现有来源工作簿字段保留：API 结果使用固定来源标签 `API:EBC+Sina`，来源摘要字段保存本行标准化输入、规则版本和参数版本的 SHA-256，而不是伪造 Excel 文件。

### 7.3 同步运行日志表

新增 `iron_ore_basis_sync_runs`，建议字段：

- `id`；
- `trigger_type`：`scheduled_2130`、`retry_0930`、`retry_1030`、`startup_catchup`、`manual`；
- `target_start_date`、`target_end_date`；
- `status`：`running`、`success`、`partial`、`failed`、`skipped`；
- `source_point_count`、`result_insert_count`、`duplicate_skip_count`、`missing_combo_count`；
- `error_code`、`error_summary`；
- `started_at`、`finished_at`。

日志只保存摘要和可排障错误，不保存账号、密码、令牌或完整第三方响应。

## 8. 完整性、事务与异常处理

### 8.1 写入前校验

每个待生成组合必须同时满足：

- 是配置范围内的已完成交易日；
- `I0` 存在该日收盘价；
- 当前规则包配置的对应 EBC 指标存在该日价格；
- 能找到当前有效规则包参数；
- 业务唯一键无重复；
- 价格为有效正数，计算结果通过现有质量边界。

满足全部条件的组合进入本轮写入；缺失 EBC 的组合跳过并等待重试，不阻塞同日其他完整组合。完整性检查的“预期指标集合”以当前有效规则包中明确配置的映射为准，不简单假设 15 个品种 × 8 个港口必然每天都有 120 条。

任一轮有完整组合写入但仍存在缺失时，运行日志可记录为 `partial`；第三次仍缺失的组合继续跳过，不生成用户可见异常。`I0` 整日缺失时没有任何组合具备完整输入，因此本轮不写结果并按相同次数规则重试。

### 8.2 原子写入

网络请求和计算在事务外完成。对本轮已经完整的组合校验通过后，在一个数据库事务中：

1. 写入新的原始来源点；
2. 写入或跳过现有期现结果；
3. 写入一一对应的计算明细；
4. 写入成功运行日志。

任一步失败则本轮全部写入整体回滚，不留下只有原始点、只有结果或缺少明细的半条业务记录。缺失组合本来就不进入事务，不影响完整组合发布。失败日志在回滚后以独立短事务记录。

### 8.3 历史来源变化

如果同一来源唯一键已经存在但新值不同：

- 不覆盖原值；
- 不重算历史基差；
- 本次任务记录来源数据差异；
- 日志记录来源、指标、日期、旧值和新值；
- 后续由人工判断是否需要创建新的业务规则或执行单独的数据修复任务。

### 8.4 第三方失败

- 登录失败、令牌无效、限流、超时、返回格式变化分别使用稳定错误码；
- 21:30 失败后由次日 09:30 和 10:30 重试；
- 10:30 后仍有缺失则继续跳过，下一次启动补检或后续日常回看继续处理；
- 页面继续展示已经成功写入的完整组合，不生成或返回缺失输入的结果；
- 环境开关可立即关闭自动同步而不影响网站只读查询。

### 8.5 页面最新数据日期

- 最新日期取 `iron_ore_basis_results.business_date` 的最大值；
- 期现数据管理和期现数据展示顶部统一显示“最新数据日期：YYYY-MM-DD”；
- 任何完整组合成功写入新交易日后，最新日期随结果表自动前进；
- 如果目标日期没有任何完整组合写入，最新日期保持不变；
- 初版不显示正常/异常状态、缺失组合数量或详情，也不提供重试、修改和处理按钮。

## 9. 凭据与环境隔离

### 9.1 Render Secret

预期环境变量：

- `EBC_ACCOUNT`；
- `EBC_PASSWORD`；
- `IRON_ORE_BASIS_AUTO_SYNC_ENABLED`；
- 必要时增加不含敏感信息的时区或运行模式配置。

账号和密码只保存在 Render Secret 中，不写入代码、Git、数据库、测试夹具、文档或发布记录。由于旧密码曾在聊天中出现，Production 配置前必须先更换密码，再把新密码直接录入 Render Secret。

### 9.2 登录与令牌

- 每次同步任务开始时使用 Render Secret 登录；
- 取得的 EBC 访问令牌只保存在当前进程内存中；
- 任务结束后丢弃，不写文件、不写数据库、不写日志；
- 新浪接口无需账号或密钥。

### 9.3 Staging 与 Production

- Staging 的 `IRON_ORE_BASIS_AUTO_SYNC_ENABLED` 默认关闭，只通过受控命令或内部调用人工触发；
- Production 只有在测试版验收和用户 Gate B 确认后才配置凭据、执行数据库迁移并开启自动调度；
- 不允许 Staging 和 Production 同时自动登录同一 EBC 账号；
- Production 数据库变更前必须记录备份、迁移和回滚方案。

### 9.4 数据库访问控制

- `iron_ore_basis_source_points` 和 `iron_ore_basis_sync_runs` 沿用现有期现表的安全模式；
- 两张新表在 `public` schema 中启用 RLS，但不为 `anon` 或 `authenticated` 创建直接访问策略；
- 撤销 `anon`、`authenticated` 对新表及相关序列的直接权限；
- 浏览器只能通过现有 FastAPI 权限体系读取“最新数据日期”，不能直接访问原始来源点和同步日志；
- Staging 迁移后和 Production 发布后分别检查 Supabase Security Advisor、RLS、表授权和序列授权。

## 10. 测试与验收

### 10.1 计算回放

- 从已验收工作簿“计算明细”读取每行的湿吨价、参数和 `I0` 收盘价；
- 使用新 Python 计算引擎回放全部 60,424 条历史明细；
- 干吨价、五项调整、质量调整合计、标准化现货价和基差均与工作簿结果一致；
- 浮点比较使用明确的小误差阈值，不以页面四舍五入值替代原始结果；
- 2026-07-10 昆巴粉/京唐港基准必须复现：湿吨价 746、标准化现货价约 739.489711934156、`I0` 收盘价 751.5、基差约 -12.010288065844。

### 10.2 自动化测试

至少覆盖：

- EBC 和新浪响应标准化；
- 15 个品种映射和特殊品种规则；
- 五项质量升贴水边界；
- 干湿吨和基差公式；
- 目标交易日、周末、节假日和跨年；
- 21:30、09:30、10:30 的目标日期选择；
- 启动补检从最新结果日后开始；
- 同一任务运行两次不增加记录；
- 缺失 EBC 的组合不写结果，但不阻塞同日其他完整组合；
- 三次尝试后仍缺失的组合继续跳过，后续数据恢复后能够补齐；
- 页面最新数据日期在新交易日有成功结果时前进，完全没有成功结果时保持不变；
- 历史来源变化只记录、不覆盖；
- 数据库事务中途失败整体回滚；
- 并发任务只有一个取得运行锁；
- 新规则从下一个交易日起生效且不改历史；
- 日志不含密码和令牌。
- 新增两张表启用 RLS，`anon`、`authenticated` 无直接表或序列权限；

### 10.3 Staging 真实验收

1. 在 Staging 人工触发，不开启自动定时；
2. 验证 Render 能登录 EBC、查询目标日频指标并抓取新浪 `I0`；
3. 从 2026-07-13 开始补齐到最近已完成交易日；
4. 再执行一次相同任务，确认结果和明细数量不增加；
5. 在真实 Staging 页面打开“期现数据管理”和“期现数据展示”；
6. 确认 2026-07-10 历史段与 2026-07-13 API 段连续显示；
7. 核对新增日期的管理表、最优仓单和基差图；
8. 确认两个期现页面顶部显示相同的最新数据日期；
9. 模拟目标日完全无可写结果，确认最新数据日期保持不变；
10. 检查浏览器控制台、后端日志和数据库一致性；
11. 确认 Staging 自动同步开关仍为关闭。

### 10.4 Production 验收

Production 仅在单独确认后执行：

- 记录待发布 commit、数据库备份和迁移回滚方案；
- 配置轮换后的 EBC Secret；
- 执行加法式迁移并验证现有 60,424 条历史数据未改变；
- 开启自动同步；
- 验证健康接口、一个浏览器可见页面和最近目标交易日；
- 更新版本记录，但不写入任何凭据或敏感接口信息。

## 11. 回滚与停止机制

- 将 `IRON_ORE_BASIS_AUTO_SYNC_ENABLED` 设为关闭，可立即停止后续自动任务；
- 调度器代码可以随应用版本回滚，现有页面继续读取已经写入的完整结果；
- 两张新增表和所需索引采用加法式迁移，代码回滚时不要求立即删除；
- API 增量结果有来源摘要和规则版本，可按明确日期范围执行单独的数据修复，但本功能不提供网页删除或编辑入口；
- 不自动删除历史 Excel 数据或 API 数据。

## 12. 明确不包含的范围

- 不建设新的网页、独立数据展示系统或移动端应用；
- 不新增 Render 服务或 Cron Job；
- 不允许网页直接请求 EBC 或新浪；
- 不建设规则编辑页面、年度参数管理页面或数据修订页面；
- 不自动覆盖 EBC、`I0` 或历史期现结果；
- 不回填过去三年的原始来源点表；
- 除顶部增加只读“最新数据日期”外，不改变现有期现页面筛选、图表和权限模型；
- 初版不识别停更指标、不生成数据同步异常提醒；
- 不执行任何金融交易、下单或账户资金操作。

## 13. 成功标准

本功能完成的判定条件：

1. 现有 2024-01-02 至 2026-07-10 历史结果保持不变；
2. 2026-07-13 起每个可用交易日由 API 自动生成与现行规则一致的期现基差；
3. 历史段和 API 段在现有管理表与图表中连续展示；
4. 重复运行不产生重复，缺失组合不阻塞完整组合，单条结果和明细不会半写入，历史来源变化不被静默覆盖；
5. Mac 关机且无人打开网页时，Production Render 仍能按计划更新；
6. Staging 与 Production 隔离，凭据不进入代码、日志和数据库；
7. 新业务规则只从用户指定启用后的下一个交易日起生效，不修改历史结果。
8. 两个期现页面清楚显示结果表中的最新业务日期，让用户能够判断数据更新到了哪一天。
