# 人员名单导入与默认密码规则设计

## 目标

基于 `/Users/wangjingze/建龙/人员名单.xlsx` 在现有用户权限体系上补充“公司领导”部门、按身份和部门生成初始密码，并支持名单中的受控个人例外。先在 Staging 完成 dry-run、建号和验收；Production 建号继续要求单独确认。

## 名单输入

当前工作簿只有一个页签，表头为 `姓名、部门、用户类型`，共 26 名人员。导入器兼容三列名单，不要求补充登录账号列；登录账号统一由姓名全拼小写生成。读取时清理单元格首尾空格，因此 `用户 ` 与 `用户` 视为同一类型。

允许部门扩展为：`贸易处、期货组、财企处、资金处、管理部门、公司领导`。名单中的三名公司领导保留“公司领导”部门，不映射为管理部门。

## 密码生成规则

统一由后端密码策略函数生成，页面预览、用户创建、管理员默认重置和名单 dry-run 共用同一实现。`username` 指姓名全拼小写生成的登录账号。

规则按以下优先级执行：

1. 曹骧为长期个人例外：密码为 `username`，即 `caoxiang`。
2. 其他领导：密码为 `username + 12345`。
3. 贸易处、期货组普通用户：密码为 `username`。
4. 财企处、资金处普通用户：密码为 `username + 123`。
5. 其它允许组合使用现有兼容回退 `username + 123`，但名单 dry-run 会明确标记为回退规则，避免静默采用。

普通新账号仍设置 `password_change_recommended = 1`。用户自助改密的至少 8 位限制只约束新密码，不阻止按上述规则创建较短的初始密码。

## 王景泽现有账号

王景泽不在名单中，不作为新用户创建。Staging 验收时单独将现有账号调整为：

- 部门：`管理部门`
- 用户类型：`管理员`
- 密码：`admin`
- `password_change_recommended = 0`，不显示改密提醒

该密码只通过管理员受控设置流程写入哈希，不写入审计日志、版本记录或通用密码规则常量。后续若管理员主动执行默认密码重置，则按通用兼容回退规则生成新临时密码并恢复改密提醒；本次设置不改变这一行为。

## 权限规则

“公司领导”用户继续使用领导规则：默认获得全部已上线业务模块查看权，不获得日常操作、敏感操作或后台管理权限。

龙云飞先按贸易处普通用户生成部门默认权限，再增加个人例外：

- `订单融资进度`：日常操作
- `融资资金监控`：日常操作
- 两个子模块的敏感操作均关闭

其它名单用户只生成部门或领导默认权限快照，不批量授予敏感权限。

## 接口与导入流程

1. 扩展后端部门白名单和前端部门下拉框，加入“公司领导”。
2. 将临时密码生成收敛到单一函数，预览、创建和默认重置均调用该函数。
3. 名单脚本支持当前三列表头，也继续兼容可选的 `登录账号（可空）` 列。
4. dry-run 对每一行输出姓名、账号、部门、用户类型、密码规则、权限数量和个人例外差异；非法部门、非法类型、拼音账号冲突或管理员部门错误会中止整批。
5. 正式写入复用用户创建 API。龙云飞的两个订单融资权限随最终权限快照事务写入。
6. 王景泽的资料和密码调整作为独立步骤执行，不能依赖名单新增流程。

## 安全与审计

- 密码只在创建或重置响应和受控 dry-run 中展示，不写入应用日志、操作日志和发布记录。
- 审计记录只保存“创建用户、设置权限、管理员设置密码、修改用户类型/部门”等脱敏摘要。
- Staging 建号前复用现有认证表备份方案；Production 必须重新备份，不能直接复用 Staging 备份。
- Production 操作必须先展示名单 dry-run 和账号冲突结果，并在用户明确确认后执行。

## 验收标准

- 工作簿 26 人全部通过 dry-run，首尾空格被正确归一，账号无冲突。
- 三名公司领导保留“公司领导”部门并只拥有 9 个已上线业务模块查看权。
- 曹骧、其他领导、贸易处、期货组、财企处、资金处的密码预览分别符合规则。
- 龙云飞可查看和日常操作两个订单融资子模块，但导入、导出、删除和管理动作返回 403。
- 王景泽显示为“管理部门 / 管理员”，可用指定密码登录且不显示改密提醒。
- 测试账号登录、菜单、按钮和真实后端接口均验证；日志中不得出现任何明文密码。
- Staging 验收完成后更新版本记录；Production 保持不变，直至单独确认。

## 回滚

代码回滚恢复旧部门白名单和 `username + 123` 统一临时密码规则。Staging 账号及权限按本次建号批次精确删除或从认证相关备份恢复，不触碰任何业务台账表。
