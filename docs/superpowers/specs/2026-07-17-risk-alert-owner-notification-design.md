# 风险预警设置人通知与页面布局调整设计

## 背景与目标

当前风险预警会记录设置人姓名，但通知收件人仍依赖可手工输入的 `reminder_users`。当该字段留空时，系统将其解释为“提醒所有人”，导致无关用户也收到预警通知。同时，预警管理页的规则和历史为左右布局，历史时间直接展示数据库原始字符串。

本次目标是：

1. 每条预警自动绑定创建它的登录账号，仅设置人本人收到该预警通知。
2. 预警规则在上、预警历史在下，改为上下布局。
3. 预警时间统一显示为 `YYYY-MM-DD HH:mm:ss`，不显示小数秒或时区尾串。

## 业务规则

### 设置人与提醒对象

- 新建预警时，后端以当前已认证用户作为唯一设置人，保存稳定的用户 ID，并继续保留设置人姓名供界面显示。
- 前端不再提供“提醒人”自由输入框，不允许通过空值或手工姓名建立广播通知。
- 预警规则列表将“提醒人”列改为“设置人”，显示创建时保存的姓名。
- 编辑、启停或模拟触发不改变原设置人；管理员或其他有权用户的编辑操作不会将预警转移给编辑者。
- 本次不提供委托他人、多收件人或预警所有权转移功能。

### 通知隔离与已读

- 通知列表只返回当前用户所属预警的未读触发记录。
- 通知数字角标仅统计当前用户可接收的未读记录。
- 单条标记已读时，后端必须校验该触发记录属于当前用户的预警；不属于当前用户时不得更新。
- “全部已读”只更新当前用户所属预警的未读记录，不得更新其他用户的记录。
- 预警管理页的共享规则列表和预警历史查看范围保持现有模块权限逻辑；本次仅收紧顶部通知投递与通知已读写入边界。

### 历史数据兼容

- `alert_settings` 增加可空的设置人用户 ID 字段，同时保留现有 `creator` 文本字段。
- 对旧预警，仅在 `creator` 姓名能唯一匹配到现有用户时回填用户 ID。
- 无设置人、设置人已删除或姓名无法唯一匹配的旧预警不再广播；它们仍保留在预警管理列表和历史中，但不会投递给无关用户。
- 不删除现有 `reminder_users` 字段，避免破坏性迁移；新通知逻辑不再依赖它。

## 页面布局与时间显示

- 仅为 `riskAlertPage` 增加专用的纵向容器样式，不修改其他使用 `.content-grid` 的页面。
- “预警规则”位于上方，“预警历史”位于下方，两区块各自保持现有表格滚动和操作能力。
- 桌面和窄屏均使用上下顺序，不再因屏幕宽度切换为左右布局。
- 前端使用专用格式化函数处理预警历史时间：将 `T` 转换为空格，仅保留日期和时分秒，不显示小数秒和时区后缀。本次只收窄展示精度，不改变存储时间或业务时区口径。

## 数据与接口边界

- 数据库变更为单表增加一个可空字段和幂等回填，不改变预警触发、阈值计算、扫描频率或历史记录结构。
- 创建预警接口不信任前端传入的提醒人，设置人始终来自当前服务端认证会话。
- 预警查询、创建、编辑、启停、删除和模拟触发的现有权限等级保持不变。
- 不操作 Production 数据库、Production Render、`main` 分支或正式环境变量。

## 测试与验收

### 自动化断言

1. 用户甲创建预警后，规则保存甲的用户 ID 和姓名。
2. 该预警触发后，甲的通知列表包含该记录，用户乙的通知列表不包含。
3. 用户乙不能将甲的触发记录标记已读。
4. 甲执行“全部已读”时，只更新甲所属预警的触发记录。
5. 旧预警姓名唯一匹配时可回填账号；无法归属时不向任何人广播。
6. 风险预警页使用专用上下布局，其他 `.content-grid` 页面样式不变。
7. 带小数秒或时区尾串的预警时间只渲染到秒。

### Staging 真实界面验收

1. 使用两个有风险预警权限的 Staging 测试账号，甲新建并模拟触发一条预警。
2. 甲的顶部通知显示该预警，乙的通知数不增加且列表不显示该记录。
3. 甲标记已读后通知消失；乙无法对甲的通知执行已读写入。
4. 风险预警页显示上方预警规则、下方预警历史，规则列表显示“设置人”，新建对话框无“提醒人”输入框。
5. 预警历史时间显示为 `YYYY-MM-DD HH:mm:ss`，页面控制台无新增错误或警告。
6. 验收完成后精确删除本次创建的 Staging 临时预警及其历史记录。

## 非目标

- 不修改风险指标计算、预警阈值、扫描频率、触发去重或行情数据源。
- 不改变风险预警模块的规则和历史共享查看权限。
- 不新增短信、邮件、企业微信、WPS 或其他外部消息渠道。
- 不新增多人订阅、代理设置或所有权转移。
- 不调整其他业务页布局、通知或时间格式。

## 回滚边界

- 代码回滚后可保留新增的可空用户 ID 字段，不需要破坏性删列。
- 布局和时间格式化均为前端局部变更，可随功能提交一并回滚。
- 本次 Staging 验收仅写入可精确删除的临时预警数据，不修改 Production 数据。
