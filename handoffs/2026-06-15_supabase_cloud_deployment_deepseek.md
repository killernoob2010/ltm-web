# 给执行模型的说明

你将接手“轻量化交易管理系统 Web”的云端部署改造任务。
请严格按本文执行，不要自行扩大范围。
任何会破坏现有本地可用版本的操作，必须先备份。

本文是给 DeepSeek 或其他执行模型使用的任务包。目标是把当前本地运行的 Web 系统，改造成 Mac 关机后仍可通过网址登录使用的云端版本。

## 1. 任务背景

当前系统已经从桌面版逐步迁移为 Web 版，但正式运行仍依赖本地 Mac：

- 本地 Mac 运行 Python/FastAPI 后端。
- 本地 SQLite 文件 `backend/data/app.db` 保存业务数据。
- 用户通过本地地址访问系统。

用户的最终目标是：

```text
Mac 只负责开发和维护
云端负责正式运行
任意电脑通过网址登录使用
```

因此，本次任务不是单纯“换数据库”，而是完成一版低成本云端验证方案：

```text
任意电脑浏览器
  -> 云端网址
  -> 云端 FastAPI 后端
  -> Supabase PostgreSQL 统一数据库
```

### 6.3 设计 PostgreSQL schema

根据当前 SQLite schema 生成 PostgreSQL 建表 SQL。

类型建议：

```text
SQLite INTEGER PRIMARY KEY AUTOINCREMENT
  -> PostgreSQL GENERATED BY DEFAULT AS IDENTITY

TEXT 日期时间
  -> timestamptz 或 text

REAL 金额/价格/汇率/手续费
  -> numeric

INTEGER 布尔值
  -> boolean 或继续 integer 兼容
```

第一阶段为了降低风险，可以保守处理：

- 主键保留整数 ID。
- `can_view`、`can_edit`、`is_closed` 可先保留整数 `0/1`，减少业务代码改动。
- 日期字段如当前代码强依赖字符串格式，可先保留 `text`，后续再统一时间类型。
- 金额和价格字段优先改为 `numeric`，但需要确保 Python 返回 JSON 时能正确序列化。

必须保留当前唯一约束和索引：

```text

calculated_data: UNIQUE(info_type, year, month, calc_date)
daily_prices: UNIQUE(info_type, contract_code, calc_date)
strategy_groups: UNIQUE(group_name)
```

必须保留关键索引：


### 6.6 改造数据库连接层

重点改造 `backend/app/db.py`。

目标：

- 云端正式运行时通过 `DATABASE_URL` 连接 Supabase PostgreSQL。
- 本地开发可以继续连接 SQLite，或也连接 Supabase 测试库。
- 数据库连接逻辑集中在一个地方，不要在业务代码里到处写连接串。

建议方案：

```text
如果 DATABASE_URL 以 postgresql:// 或 postgres:// 开头：
  使用 PostgreSQL 连接
否则：
  保留 SQLite 本地连接
```

具体库选择：

- 优先选最小改动方案。
- 可使用 `psycopg` 或 SQLAlchemy。
- 如果直接用 `psycopg`，需要处理 row 返回格式，让现有代码能继续用类似 `row["field"]` 的方式访问。
- 如果使用 SQLAlchemy，注意不要为了迁移做大规模 ORM 重写。

硬性要求：

- 不能把真实数据库密码写进 `db.py`。
- 不能让前端直接连接 Supabase 数据库。
- 不能把 Supabase service role key 暴露给浏览器。

### 6.7 修复 SQL 兼容问题

当前代码可能存在 SQLite 特有写法，需要逐项处理。

重点检查：

```text
? 参数占位符
INSERT OR REPLACE
INSERT OR IGNORE
lastrowid
CURRENT_TIMESTAMP
month IS ?
PRAGMA table_info
executescript
AUTOINCREMENT
```

处理建议：

- 参数占位符按所选数据库库的要求统一处理。
- `INSERT OR REPLACE` 在 PostgreSQL 中应改为 `INSERT ... ON CONFLICT ... DO UPDATE`。
- `INSERT OR IGNORE` 在 PostgreSQL 中应改为 `INSERT ... ON CONFLICT DO NOTHING`。
- `lastrowid` 改为 `RETURNING id`。
- `PRAGMA table_info` 只适用于 SQLite，PostgreSQL 需要改成 `information_schema` 或独立迁移机制。
- SQLite 的 `month IS ?` 不要直接照搬到 PostgreSQL。需要显式处理 null：

```text
month = value
或 value 为 null 时 month IS NULL
```

### 6.11 安全加固

公网访问前最低安全要求：

- 默认管理员密码不能继续使用 `admin`。
- 密码哈希不应长期使用普通 SHA-256；建议升级为 bcrypt 或 Argon2。
- 如果第一阶段暂不重做密码体系，至少必须强制修改默认管理员密码。
- 登录 token 应有过期时间。
- 管理员接口必须校验管理员权限。
- 写入接口必须校验登录状态和模块编辑权限。
- 数据库连接串只放环境变量。
- `.env` 不得提交。
- CORS 不应允许任意来源长期访问。

第一阶段如果为了降低风险不完整重做认证，必须在交付说明中明确写出剩余风险。


如果云端部署失败：

1. 不删除原始 `app.db`。
2. 不覆盖本地可用版本。
3. 本地继续使用 `start_dev.sh` 或 `start_prod.sh`。
4. 保留 Supabase 项目用于排查。
5. 从备份目录恢复代码和数据。

如果数据迁移失败：

1. 清空或重建 Supabase 测试表。
2. 重新从原始 SQLite 迁移。
3. 不直接修改原始 SQLite 文件。
4. 记录失败表、失败 SQL、失败原因。

## 9. 风险点

主要风险：

- SQLite 和 PostgreSQL SQL 方言不同。
- `INSERT OR REPLACE`、`lastrowid`、`PRAGMA` 等写法不能直接用于 PostgreSQL。
- 免费云服务可能休眠，首次访问可能较慢。
- 外部行情接口在云端网络环境下可能和本地表现不同。
- 当前认证安全性不足，不适合长期裸露在公网。
- 如果忘记配置环境变量，云端后端可能启动失败。
- 如果真实密钥被提交到代码仓库，会带来严重安全风险。

处理原则：

- 先备份，再改造。
- 先本地连接 Supabase 验证，再部署云端。
- 每一步都保留可回滚状态。
- 发现不确定点先记录，不要猜测式重构。

每份交接文档建议使用固定结构：

```text
1. 任务背景
2. 当前项目状态
3. 最终目标
4. 明确不做什么
5. 当前代码/数据位置
6. 详细执行步骤
7. 验收标准
8. 回滚方案
9. 风险点
10. 给执行模型的硬性要求
```

`handoffs/` 只存可执行交接文档：

- 不存运行代码。
- 不存真实密钥。
- 不存数据库连接串。
- 不替代 `docs/` 中的长期架构文档。
