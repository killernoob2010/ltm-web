# 风险预警历史按规则分组与个人工作区设计

## 背景

当前风险预警历史按触发记录平铺展示。前端默认请求最多 200 条记录，再直接生成表格。规则较多、同一规则多次触发后，用户难以判断某条历史属于哪条规则，首次加载也会随历史量增长。

本次将风险预警改造成按账号隔离的个人预警工作区，并为管理员保留全局管理能力。预警历史以规则为第一层级，通过服务端分页返回汇总；用户展开规则后，再按需加载该规则的触发详情。

## 目标

1. 普通用户只能查看和管理自己创建的规则及历史。
2. 管理员可以查看和管理全部规则及历史，但不会收到其他人规则的预警通知。
3. 预警历史只展示至少触发过一次的规则。
4. 历史第一层按规则汇总并分页，第二层展开后按需加载触发详情。
5. 删除已经触发过的规则时保留历史，历史明确标识规则已删除。
6. 汇总层允许删除该规则的全部历史。
7. 管理员修改他人规则或历史时留下可审计的操作日志。
8. 所有时间继续只显示到秒。

## 非目标

- 不改变预警计算公式、阈值判断或后台自动扫描频率。
- 不增加多人共同接收同一规则预警的能力。
- 不增加单条历史永久删除入口。
- 不增加历史导出、图表或跨规则分析。
- 不改变其他业务模块的权限模型。

## 权限模型

### 普通用户

只要拥有风险预警模块的查看权限，即可执行本模块全部日常操作：

- 新增规则；
- 编辑自己的规则；
- 启停自己的规则；
- 模拟触发自己的规则；
- 删除自己的规则；
- 查看自己的规则汇总和触发详情；
- 标记自己的预警详情为已读；
- 删除自己的整组历史；
- 对自己的规则执行批量启停、批量删除和手动扫描。

所有写接口必须在后端校验 `creator_user_id`，不能依赖前端隐藏按钮实现隔离。

### 管理员

管理员可以：

- 查看全部人员的规则、历史汇总和触发详情；
- 按设置人筛选规则和历史；
- 编辑、启停、模拟或删除任意人员的规则；
- 标记任意人员的预警详情为已读；
- 删除任意规则的整组历史；
- 执行全局手动扫描。

管理员操作他人数据时必须写入操作日志。日志至少记录：

- 操作人；
- 原设置人；
- 规则 ID；
- 操作类型；
- 删除历史时的记录数量；
- 操作时间。

管理员只拥有管理可见性，不进入其他人员的通知接收范围。

### 通知

- 未读通知始终按 `creator_user_id` 发送给规则设置人。
- 管理员不会因为可查看全部数据而收到其他人员的通知。
- 单条已读和全部已读继续按规则归属处理；管理员代操作时按明确选择的目标记录执行并写日志。

## 规则生命周期

为保留历史与规则之间的稳定关联，已触发规则不执行立即物理删除。

### 从未触发的规则

删除时直接物理删除 `alert_settings` 记录。

### 已触发的规则

删除时：

1. 停用规则；
2. 写入归档标记和归档时间；
3. 从上方预警规则管理列表隐藏；
4. 保留规则记录及全部 `alert_history`；
5. 历史汇总行显示“规则已删除”。

### 删除整组历史

汇总层的“删除历史记录”按钮删除该规则的全部 `alert_history`：

- 规则仍处于活跃或停用状态时，只删除历史，规则继续保留；
- 规则已经归档时，删除全部历史后同时物理删除归档规则；
- 删除操作必须使用确认对话框，显示规则名称和将删除的历史数量；
- 操作完成后记录操作日志。

## 历史汇总

历史第一层以 `alert_id` 为分组键，只返回至少存在一条历史的规则。

默认每页 10 条规则，并支持常用页码切换。汇总行按最近预警时间倒序排列。

每条汇总显示：

- 指标；
- 合约；
- 设置人；
- 规则状态，归档规则显示“规则已删除”；
- 最近一次当前值；
- 最近一次触发时的预警值；
- 预警总次数；
- 未读次数；
- 最近一次预警方向；
- 最近一次预警时间；
- 展开/收起操作；
- 删除历史记录操作。

最近当前值、预警值、方向和时间必须来自最近一条 `alert_history`，不能使用规则后来编辑后的当前阈值替代历史事实。

普通用户只能查询自己的汇总。管理员默认查询全部汇总，并可以按设置人筛选。

## 触发详情

用户展开一条规则时才请求该规则的历史详情。

- 默认加载最近 20 条；
- 按 `alert_time DESC, id DESC` 排序；
- 超过 20 条时显示“加载更多”；
- 每次继续加载下一批，不重复已有记录；
- 收起后保留当前页内已加载内容，切换规则汇总页时清空展开状态。

详情字段：

- 预警时间；
- 当前值；
- 触发时预警值；
- 方向；
- 已读状态；
- 标记已读操作。

普通用户不能请求其他人员规则的详情。管理员可以请求全部规则详情。

## 页面交互

预警历史区域从平铺表格改为可展开的规则汇总列表。

### 汇总层

- 点击汇总行或“展开”按钮显示详情；
- 同一页允许展开多条规则；
- “删除历史记录”位于汇总行操作区；
- 删除前显示规则名称、设置人和历史数量；
- 管理员查看全部数据时显示设置人筛选；
- 普通用户不显示无意义的设置人筛选。

### 分页

- 汇总分页默认 10 条规则；
- 页面显示总规则数、当前页和总页数；
- 支持上一页、下一页和页码切换；
- 详情采用每次 20 条的“加载更多”，避免嵌套页码造成页面过于复杂；
- 切换汇总页时收起所有规则并清理详情缓存。

### 空状态

- 当前用户没有任何触发历史时显示“暂无预警历史”；
- 管理员筛选某个设置人且无结果时显示“该设置人暂无预警历史”；
- 删除最后一组历史后重新计算总页数，必要时返回上一有效页。

## 后端接口

### 规则列表

现有规则列表接口增加归属过滤：

- 普通用户只返回 `creator_user_id = 当前用户 ID` 且未归档的规则；
- 管理员返回全部未归档规则，并支持设置人筛选；
- 批量操作和单条操作使用同一归属规则。

### 历史规则汇总

新增按规则汇总的服务端分页接口，接收：

- `limit`，默认 10，设置合理上限；
- `offset`；
- 管理员可选的 `creator_user_id` 筛选。

返回：

- `items`：规则汇总；
- `pagination.total`：至少触发过一次的规则数量；
- `pagination.limit`；
- `pagination.offset`；
- `pagination.has_more`。

聚合统计和最近一条触发记录必须由数据库查询完成，不能先加载固定数量历史再在应用层统计。

### 单规则历史详情

新增单规则详情分页接口，路径包含 `alert_id`，接收：

- `limit`，默认 20；
- `offset`。

接口在返回前校验：

- 规则存在；
- 普通用户是设置人；
- 管理员拥有全局管理能力。

### 删除整组历史

新增按规则删除历史接口：

1. 校验归属或管理员身份；
2. 读取待删除数量；
3. 在事务内删除该规则全部历史；
4. 若规则已归档则同时物理删除规则；
5. 返回删除数量；
6. 写入操作日志。

### 规则操作

新增、编辑、启停、模拟、删除、批量操作和手动扫描均按以下原则调整：

- 普通用户只操作自己的规则；
- 管理员可操作全部规则；
- 管理员操作他人规则时写操作日志；
- 普通用户手动扫描只扫描自己的有效规则；
- 管理员手动扫描全部有效规则；
- 后台定时扫描继续扫描全部有效规则。

## 数据库调整

`alert_settings` 增加可空归档时间字段，例如 `archived_at`。

- `archived_at IS NULL`：正常规则；
- `archived_at IS NOT NULL`：已删除但因存在历史而保留的归档规则。

新增适用于 PostgreSQL 和 SQLite 的索引：

```text
alert_history(alert_id, alert_time DESC, id DESC)
```

如果汇总查询性能需要，再增加与设置人和归档状态相关的规则索引；不在没有查询证据时增加额外索引。

迁移必须幂等。现有规则默认 `archived_at = NULL`，不改变已有规则状态和历史数据。

## 错误处理

- 普通用户请求他人规则时统一返回 404，避免泄露规则是否存在。
- 管理员操作不存在的规则返回 404。
- 删除历史时规则不存在或已经没有历史，返回明确的无可删除记录结果，不产生伪成功日志。
- 分页参数自动限制在安全范围。
- 删除历史和清理归档规则必须处于同一事务。
- 前端请求失败时保留当前汇总列表并显示错误，不清空已加载数据。

## 测试与验收

### 后端自动化

覆盖：

- 两个普通用户只能看到自己的规则和汇总；
- 普通用户无法访问、编辑、模拟、删除或扫描他人规则；
- 管理员能看到并筛选全部规则和历史；
- 管理员操作他人规则产生操作日志；
- 管理员不进入其他人的通知结果；
- 汇总统计、最近值、方向、未读数和排序正确；
- 汇总按规则分页而不是按历史条数分页；
- 详情每次返回 20 条并正确继续加载；
- 删除未触发规则执行物理删除；
- 删除已触发规则执行归档且历史保留；
- 删除活跃规则历史后规则保留；
- 删除归档规则历史后规则被物理清理；
- SQLite 与 PostgreSQL 迁移和索引创建幂等。

### 前端自动化

覆盖：

- 汇总行字段和展开入口；
- 管理员设置人筛选；
- 普通用户不显示设置人筛选；
- 详情展开时才请求接口；
- 加载更多追加而非覆盖；
- 汇总分页切换清空展开状态；
- 删除历史确认信息包含规则和数量；
- 空状态和最后一页回退；
- 时间显示到秒。

### Staging 验收

使用两个普通账号和一个管理员账号验证：

1. 两个普通账号分别创建并多次模拟触发自己的规则；
2. 各普通账号只能看到自己的规则和历史；
3. 管理员看到双方数据并按设置人筛选；
4. 管理员修改他人规则，操作日志产生对应记录；
5. 管理员不会收到其他人员规则的通知；
6. 汇总分页、展开、加载更多和最近值统计正确；
7. 删除已触发规则后历史保留并显示“规则已删除”；
8. 删除整组历史后无临时数据残留；
9. 页面标题、静态资源和控制台状态正常。

验收使用带唯一标记的临时规则。测试完成后精确清理临时规则、历史和相关临时账号；操作日志作为审计记录保留。

## 发布与回滚

- 先完成本地和 Staging 验收。
- Production 发布需要单独 Gate B 确认。
- 数据库迁移前按项目规范备份目标环境。
- 代码回滚时可保留 `archived_at` 和新增索引。
- 如果需要回滚归档行为，已归档规则默认保留，不能直接删除历史或正式业务数据。
