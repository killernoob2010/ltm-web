# 操作日志降噪、在线保留与私有归档设计

## 目标与范围

在保留关键操作追责能力的前提下，控制 `operation_logs` 的增长、查询开销和网页流量。本次先在 Staging 完成代码、数据库索引、归档工具和页面验收；不触碰 Production，不删除当前尚未达到归档条件的日志。

当前 Staging 约有 1.2 万条操作日志、表体积约 2 MB。日志页面只在管理员主动打开时请求数据，并不存在持续轮询；当前主要问题是接口每次打开都会执行 `COUNT(*)`、使用 `OFFSET` 分页、缺少查询索引，以及自动批量计算也会写入审计日志。

## 方案选择

采用“在线热数据 + 私有压缩归档”的两层方案：

1. 在线库保留最近 12 个月，并设置 20 万条软上限；页面只查询在线日志。
2. 超出条件的完整月份压缩为 gzip NDJSON，上传到 Supabase 私有 Storage，校验成功后才从在线表删除。
3. 管理员只有在主动查看或下载历史归档时才读取 Storage；日常打开日志页不会下载归档文件。

未采用以下方案：

- 永久保留全部日志：实现最简单，但表、索引和备份会持续增长。
- 只定期删除：最省空间，但失去长期追责与恢复依据。
- 立即做 PostgreSQL 分区：当前只有约 1.2 万条，远低于通常需要分区的数量级，会增加迁移和维护复杂度。

## 日志采集边界

继续记录：

- 登录成功、登录失败、退出；
- 新增、修改、删除、导入、导出、批量处理、人工刷新行情等明确用户操作；
- 用户创建、资料修改、权限修改、密码重置、账号启停和删除；
- 预警规则、交易、策略组、持仓和业务台账的写操作。

不记录：

- 页面打开、列表查询、详情查看；
- 通知轮询和其它只读轮询；
- 页面进入或恢复可见时自动触发的实时信息汇总批量计算。

实时信息汇总的手工“计算全部/刷新”与自动计算使用同一后端计算逻辑，但前端明确传递触发来源。只有手工按钮触发的批量计算写入操作日志；自动进入页面和页面恢复可见触发的不写日志。单指标手工计算仍保留日志。触发来源只决定是否记录低风险计算行为，不参与权限判断。

所有日志描述继续使用脱敏摘要，不记录明文密码、密码哈希、会话令牌、数据库连接串、Storage 密钥或完整业务文件内容。

## 在线查询与页面交互

`GET /api/operation-logs` 调整为：

- 默认 `limit=100`，最大 `200`；
- 支持操作类型、精确用户姓名、开始日期和结束日期筛选；
- 使用 `(created_at, id)` 复合游标，按 `created_at DESC, id DESC` 稳定排序；
- 每次多取一条判断 `has_more`，不再执行无条件 `COUNT(*)`；
- 返回 `logs、has_more、next_cursor`，不返回容易变慢的精确总数。

游标由服务端编码和校验，客户端不拼接 SQL。无效游标返回中文 400 错误。结束日期按用户选择日期的次日零点作为排他上界，避免漏掉当天晚间日志。

操作日志弹窗首屏只加载 100 条，新增开始/结束日期筛选和“加载更多”。再次搜索会清空旧游标和已加载行；“导出”只导出当前已经加载到页面的结果，并在按钮附近明确显示“导出当前已加载记录”，避免一次请求拉取全部日志。

## 索引

SQLite 和 PostgreSQL 均幂等创建以下索引：

- `operation_logs(created_at DESC, id DESC)`：默认时间倒序和游标翻页；
- `operation_logs(user_id, created_at DESC, id DESC)`：用户筛选和时间范围；
- `operation_logs(operation_type, created_at DESC, id DESC)`：操作类型筛选和时间范围。

用户姓名筛选先通过 `users.name` 定位用户，再以 `user_id` 查询日志，避免在日志主查询中依赖未索引的姓名连接过滤。日志写入量较低，三个针对真实查询路径的索引带来的写入成本可接受。

## 在线保留规则

归档判断同时执行两条规则：

- 时间规则：归档早于当前日期 12 个月的完整自然月；
- 数量规则：在线日志超过 20 万条时，从最老的完整自然月开始归档，直到回到 20 万条以内。

两个条件任一满足即可归档。为了保证文件边界稳定、可核验，不归档尚未结束的当前月份；如果仅当前月就超过 20 万条，在线数量可暂时高于软上限，月末后再处理，不进行高风险的月中切片删除。

归档不由网页请求触发，也不在应用启动时自动执行，避免多实例并发和启动阻塞。提供受控命令，默认只做 dry-run；后续可由 Render Cron 每月执行一次，但创建可能产生费用的定时服务不在本次自动操作范围内，需要单独确认。

## 私有归档格式与安全

归档对象使用固定私有 bucket `operation-log-archives`，文件路径按环境和月份隔离，例如：

`staging/2025/06/operation-logs-2025-06.ndjson.gz`

每行是一个完整 JSON 日志对象，gzip 压缩；不使用 CSV，避免描述中的换行、逗号或引号破坏结构。对象路径不可覆盖，重复运行遇到同路径时先校验，不直接 upsert。

压缩文件不超过 6 MB 时使用标准上传；超过 6 MB 时使用 Supabase TUS 可恢复上传，并使用官方要求的 6 MB 分块及项目的直连 Storage 主机。两种方式都禁止覆盖同路径对象。

服务端只从环境变量读取 `SUPABASE_URL` 和 `SUPABASE_SERVICE_ROLE_KEY`。service-role 仅用于后端归档与管理员下载代理，不进入 HTML、JavaScript、接口响应、日志或 Git。bucket 保持私有，不创建面向普通用户的读取策略。

新增 `operation_log_archives` 元数据表，至少保存：归档月份、对象路径、行数、首末日志时间、SHA-256、压缩字节数、创建时间和恢复时间。对象路径唯一，归档月份在同一环境唯一。

同时新增 `operation_log_archive_users` 关系表，保存每份归档涉及的用户 ID。用户物理删除保护同时检查在线日志和该关系表，避免日志归档后把有历史账号误判为可删除账号。

## 归档事务与幂等

每次受控归档按以下顺序执行：

1. 计算候选完整月份和预计行数；dry-run 到此结束，不写数据库或 Storage。
2. 以稳定顺序读取该月日志，生成 gzip NDJSON，并计算 SHA-256、行数和首末时间。
3. 上传到私有 bucket 的不可变路径。
4. 从 Storage 读取对象元信息或内容进行字节数与 SHA-256 校验。
5. 在数据库事务中写入归档元数据，并按该月时间范围删除在线日志；删除行数必须与归档行数一致，否则整个数据库事务回滚。
6. 记录一条不含归档内容的系统审计摘要。

PostgreSQL 执行时使用 advisory lock 防止两个归档任务并发处理同一月份。若上传成功但数据库事务失败，对象会保留；下次运行识别同路径、完成校验后继续元数据和删除步骤，不重复上传也不覆盖对象。任何上传或校验失败都不得删除在线日志。

提供受控恢复命令：先下载指定归档、核对 SHA-256 和行数，再在单一数据库事务中按原始日志 ID 写回 `operation_logs`。任一 ID 冲突或内容校验失败都整体回滚；成功后只标记恢复时间，不自动删除私有 Storage 对象。

## 历史归档查看

管理员可获取归档元数据列表，但列表请求不下载文件。管理员点击某个月份后，后端再次执行管理员权限校验，再从私有 Storage 以流式响应转发 gzip 文件；只有该次主动下载产生相应网络流量。

首期不在浏览器内解压和渲染整月归档，也不支持跨在线库与归档的统一全文搜索。需要查历史时下载单月 gzip 文件，避免为了偶发查询引入高流量和复杂索引系统。

## 配置与失败处理

- 没有 Storage 环境变量时，在线日志查询和写入照常工作；归档命令明确报“归档存储未配置”，不得删除数据。
- bucket 不存在或不是私有 bucket 时，正式归档停止并报错，不自动改成公开 bucket。
- 网络超时、Storage 5xx、校验不一致、数据库删除数不一致均停止当前月份，不继续处理后续月份。
- 当前 Staging 没有符合 12 个月或 20 万条条件的日志，因此验收使用临时 SQLite 测试库和伪 Storage 响应覆盖完整流程，不在 Staging 真实表中制造或删除大批日志。

## 测试与验收

### 自动化测试

- SQLite 和 PostgreSQL 初始化 SQL 均幂等创建三个查询索引和归档元数据表。
- 默认查询 100 条、最大 200 条；游标连续翻页无重复、无遗漏，并在相同时间戳下按 ID 稳定排序。
- 操作类型、用户和日期筛选组合正确；无效游标返回 400；查询路径不执行 `COUNT(*)` 或 `OFFSET`。
- 自动批量计算不写日志，手工批量计算和单指标手工计算仍写日志。
- 12 个月边界、20 万条软上限、当前月不归档和完整月份选择正确。
- dry-run 零写入；上传失败、校验失败和删除数不一致均保留在线日志。
- 成功归档的 gzip 可解压、行数和 SHA-256 一致；重复执行幂等；日志和异常输出不含 service-role 或归档正文。
- 已归档用户仍受物理删除保护；恢复时校验失败或日志 ID 冲突均零写入，成功恢复保留原始 ID 并标记恢复时间。
- 非管理员无法列出或下载归档；管理员下载采用流式响应。

### Staging 页面验收

- 打开日志弹窗只加载 100 条，筛选和“加载更多”可用，控制台无应用错误。
- 请求响应不包含精确总数，第一页不会触发全表计数。
- 日期、用户、操作类型组合筛选显示正确；导出文件只包含当前已加载行。
- 进入实时信息汇总自动计算后，日志数量不增加；点击手工刷新后增加一条脱敏日志。
- 归档列表为空时显示“暂无历史归档”，不会请求或下载 Storage 对象。

## 发布与回滚

本次只发布 Staging。数据库变更为新增索引和新增归档元数据表，不改业务台账表；在 Staging 迁移前仍按项目规则备份日志相关表。回滚代码后可保留索引和空元数据表，它们不影响旧版本；如果未来已经产生归档，回滚不得直接删除归档对象，恢复时必须先校验并将对应月份导回 `operation_logs`。

Production 上线前必须重新确认：生产备份、生产私有 bucket、服务端密钥配置、归档 dry-run 结果、回滚恢复演练和是否创建 Render 月度 Cron。
