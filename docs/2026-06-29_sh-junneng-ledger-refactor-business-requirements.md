# 上海钧能智能台账重构业务需求文档

## 0. 名称规范

本模块的公司名称必须写作“上海钧能”。“钧”为金字旁加“匀”。后续页面文案、项目文档、操作日志和需求记录都应使用“上海钧能”，不得写作“上海君能”“上海均能”或“上海𫓲能”。代码标识 `sh_junneng` 为既有模块键，继续保留。

## 1. 背景

上海钧能现有智能台账页面的目标，是把线下 Excel 台账线上化。日常业务需要持续记录三类信息：

1. 今日交易详情：当天发生了哪些开仓、平仓、手续费和成交价格。
2. 即期持仓：当前尚未平仓的持仓数量、开仓价格、现价和浮动损益。
3. 本月已结算持仓：本月已经平仓结算的交易、实际平仓价格和结算损益。

当前系统已经具备基础录入、价格更新、平仓、持仓和已结算展示能力，但数据结构仍接近线下 Excel 的一行式记录：一条记录中同时包含开仓信息和平仓信息。这种方式对完整开仓、完整平仓可以使用，但不利于未来处理部分平仓、分笔平仓和更精细的损益追溯。

因此，本次重构目标是在保留现有功能的前提下，将上海钧能智能台账改造成更适合线上持续记录的结构：开仓单独记录，平仓按每一笔成交单独记录，页面仍按业务习惯展示为今日交易、即期持仓、已结算持仓三类视图。

## 2. 当前已实现功能

当前系统中上海钧能模块已经实现以下能力，重构后必须保留：

1. 开仓录入：支持录入品种、合约、方向、手数、开仓价、手续费、交易日期等信息。
2. 平仓录入：支持对已有持仓录入平仓价格、平仓手续费、平仓日期，并将记录标记为已结算。
3. 今日交易展示：展示当天交易记录。
4. 即期持仓展示：展示未平仓记录，并根据现价计算浮动损益。
5. 已结算持仓展示：展示已平仓记录，并计算结算损益、总费用、净利润等。
6. 行情价格维护：支持合约价格更新，用于持仓浮动损益计算。
7. 月度汇总：支持按月份统计已结算数据。
8. 导出能力：支持将台账数据导出给线下模板或后续报表使用。
9. 权限与操作日志：保留现有登录、角色、操作记录等基础能力。

## 3. 当前问题

### 3.1 数据结构问题

当前一条交易记录同时承载开仓和平仓信息。对于一开一平的完整交易，结构可以满足使用；但如果出现部分平仓或多次平仓，会出现以下问题：

1. 一条开仓记录无法对应多条平仓明细。
2. 平仓数量没有独立记录，难以判断剩余持仓。
3. 已结算记录无法展示每一次真实平仓成交。
4. 损益计算依赖覆盖式字段，难以还原真实交易过程。
5. 未来导入正式台账模板时，无法稳定生成开仓行、平仓行和对应业务编码。

### 3.2 页面操作问题

当前页面更像把 Excel 表格搬到网页上，日常操作不够顺手：

1. 开仓、平仓、持仓、已结算之间的关系不够直观。
2. 平仓操作缺少“本次平仓数量”的清晰入口。
3. 用户难以一眼看出某笔持仓已经平了几次、还剩多少。
4. 已结算记录和持仓记录之间的追溯关系不够明确。

### 3.3 精度问题

线下台账会对当天同方向多笔交易做均值计算，但当价格或金额出现无法整除的情况时，如果只保存四舍五入后的价格，可能导致损益计算和真实成交金额之间存在尾差。

线上台账应尽量保存原始成交金额、成交数量、手续费等基础数据，展示时再做格式化，避免用展示值反推真实损益。

## 4. 重构目标

本次重构需要实现以下目标：

1. 保留当前所有可用功能，不减少现有业务能力。
2. 将开仓和平仓从数据结构上拆开。
3. 支持一个开仓持仓对应多笔平仓记录。
4. 即期持仓可以准确显示剩余未平数量。
5. 已结算持仓可以逐笔显示真实平仓记录。
6. 对完整开平仓场景保持简单操作，不增加不必要负担。
7. 为未来导入正式 Excel 台账模板提供完整、可追溯的数据基础。
8. 页面交互应围绕日常记账流程优化，而不是机械复刻 Excel。

## 5. 目标数据模型

### 5.1 开仓持仓表

建议新增或重构为 `sh_junneng_positions`，用于记录每一笔开仓及其当前剩余情况。

核心字段建议如下：

| 字段 | 含义 |
| --- | --- |
| id | 持仓唯一编号 |
| trade_date | 开仓日期 |
| product | 品种 |
| contract | 合约 |
| direction | 开仓方向，买开或卖开 |
| open_quantity | 原始开仓数量 |
| remaining_quantity | 当前剩余未平数量 |
| open_price | 开仓均价 |
| open_amount | 开仓成交金额，建议保留用于精确计算 |
| open_fee | 开仓手续费 |
| owner_group | 利润归属小组 |
| hedge_type | 一般套保或战略套保 |
| strategy_type | 套保策略 |
| business_code | 业务编码 |
| status | open / partial_closed / closed |
| created_at | 创建时间 |
| updated_at | 更新时间 |

说明：

1. `open_quantity` 永远表示原始开仓数量。
2. `remaining_quantity` 表示尚未平仓的数量。
3. 当 `remaining_quantity = open_quantity` 时，为未平仓。
4. 当 `0 < remaining_quantity < open_quantity` 时，为部分平仓。
5. 当 `remaining_quantity = 0` 时，为已全部平仓。

### 5.2 平仓明细表

建议新增 `sh_junneng_close_trades`，用于记录每一次平仓成交。

核心字段建议如下：

| 字段 | 含义 |
| --- | --- |
| id | 平仓记录唯一编号 |
| position_id | 对应开仓持仓编号 |
| close_date | 平仓日期 |
| close_quantity | 本次平仓数量 |
| close_price | 本次平仓价格 |
| close_amount | 平仓成交金额，建议保留用于精确计算 |
| close_fee | 平仓手续费 |
| close_sequence | 对同一开仓的第几次平仓 |
| business_code | 与对应开仓一致，必要时可追加批次后缀 |
| realized_profit | 本次实现损益 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

说明：

1. 一笔开仓可以对应多笔平仓。
2. 平仓记录必须保留对应开仓方向、开仓价格或可通过 `position_id` 追溯。
3. 损益计算以开仓方向判断：
   - 买开：平仓损益 = 平仓数量 * 合约单位 * (平仓价 - 开仓价)
   - 卖开：平仓损益 = 平仓数量 * 合约单位 * (开仓价 - 平仓价)
4. 平仓记录的业务编码默认与对应开仓一致。
5. 如未来需要区分分批平仓，可在业务编码后追加 `-1`、`-2`，但本次不强制实现。

### 5.3 行情价格表

保留当前合约现价维护能力。建议独立维护 `contract_prices` 或继续使用现有价格表。

核心要求：

1. 即期持仓使用最新合约价格计算浮动损益。
2. 已结算持仓使用真实平仓价计算结算损益，不受最新行情价格影响。
3. 价格更新时间需要可见，避免误用旧价格。

## 6. 目标页面结构

页面建议保留业务上的三大区块，但重新组织交互。

### 6.1 顶部概览区

展示当天和本月关键指标：

1. 今日开仓数量。
2. 今日平仓数量。
3. 当前未平持仓数量。
4. 本月已结算损益。
5. 当前浮动损益。
6. 数据最后更新时间。

### 6.2 今日交易详情

今日交易详情展示当天发生的所有交易动作，而不是只展示持仓记录。

建议分为两类行：

1. 开仓行：显示开仓日期、品种、合约、买开或卖开、开仓数量、开仓价、手续费、业务编码。
2. 平仓行：显示平仓日期、品种、合约、对应开仓方向、本次平仓数量、开仓价、平仓价、手续费、本次损益、业务编码。

要求：

1. 开仓和平仓在同一个今日视图中按时间顺序展示。
2. 平仓行必须能追溯到对应开仓。
3. 如果当天一笔持仓分多次平仓，应显示多条平仓行。
4. 完整开平仓的场景也应自然显示为一条开仓行加一条平仓行。

### 6.3 即期持仓

即期持仓只展示仍有剩余数量的持仓。

每行建议显示：

1. 开仓日期。
2. 品种和合约。
3. 开仓方向。
4. 原始开仓数量。
5. 已平数量。
6. 剩余数量。
7. 开仓价格。
8. 最新价格。
9. 浮动损益。
10. 业务编码。
11. 操作按钮：平仓、查看明细、编辑、撤销。

要求：

1. 完全平仓后自动从即期持仓中移除。
2. 部分平仓后仍保留在即期持仓中，并显示剩余数量。
3. 用户点击平仓时，默认平仓数量填入当前剩余数量，方便全平。
4. 用户可手动修改平仓数量，以支持部分平仓。

### 6.4 本月已结算持仓

已结算持仓展示本月已经发生的平仓明细。

每行建议显示：

1. 平仓日期。
2. 原开仓日期。
3. 品种和合约。
4. 开仓方向。
5. 本次平仓数量。
6. 开仓价格。
7. 平仓价格。
8. 开仓手续费分摊。
9. 平仓手续费。
10. 本次实现损益。
11. 业务编码。
12. 对应持仓状态：已全平或部分平仓。

要求：

1. 每一次平仓都可以单独展示。
2. 同一笔开仓多次平仓时，用户能看到多条已结算记录。
3. 如业务需要，也可以额外提供“按业务编码汇总”视图，用于对齐线下 Excel 的均值口径。

## 7. 核心业务流程

### 7.1 新增开仓

1. 用户点击新增开仓。
2. 填写交易日期、品种、合约、方向、数量、价格、手续费、套保信息、利润归属。
3. 系统生成业务编码。
4. 系统写入开仓持仓表。
5. 今日交易详情出现开仓行。
6. 即期持仓出现该笔持仓。

### 7.2 完整平仓

1. 用户在即期持仓中点击平仓。
2. 系统默认平仓数量为当前剩余数量。
3. 用户填写平仓价格、平仓手续费、平仓日期。
4. 系统新增平仓明细。
5. 系统将持仓剩余数量改为 0。
6. 持仓状态变为已全平。
7. 即期持仓不再展示该笔记录。
8. 今日交易详情出现平仓行。
9. 本月已结算持仓出现平仓记录。

### 7.3 部分平仓

1. 用户在即期持仓中点击平仓。
2. 用户将平仓数量改为小于剩余数量。
3. 系统新增平仓明细。
4. 系统扣减持仓剩余数量。
5. 持仓状态变为部分平仓。
6. 即期持仓继续展示该笔持仓，但剩余数量减少。
7. 已结算持仓展示本次平仓记录。
8. 后续再次平仓时，继续新增新的平仓明细。

### 7.4 查看明细

用户可以从即期持仓或已结算持仓打开明细，查看：

1. 原始开仓信息。
2. 历次平仓记录。
3. 每次平仓数量、价格、手续费和损益。
4. 当前剩余数量。
5. 合计已平数量和合计实现损益。

## 8. 计算规则

### 8.1 数量关系

1. 已平数量 = 该开仓下所有平仓数量合计。
2. 剩余数量 = 原始开仓数量 - 已平数量。
3. 平仓数量不得超过当前剩余数量。

### 8.2 损益计算

系统需要根据开仓方向计算损益：

1. 买开：平仓损益 = 平仓数量 * 合约单位 * (平仓价格 - 开仓价格)。
2. 卖开：平仓损益 = 平仓数量 * 合约单位 * (开仓价格 - 平仓价格)。
3. 净损益 = 平仓损益 - 开仓手续费分摊 - 平仓手续费 - 其他费用。

### 8.3 手续费分摊

如果一笔开仓分多次平仓，开仓手续费建议按平仓数量比例分摊：

开仓手续费分摊 = 开仓手续费 * 本次平仓数量 / 原始开仓数量。

### 8.4 价格精度

1. 系统保存原始价格、数量、金额和手续费。
2. 页面展示可以保留固定小数位。
3. 损益计算优先使用原始金额或高精度字段，避免使用展示值反推。

## 9. 导入导出要求

### 9.1 导入

未来如从线下台账导入，系统需要支持：

1. 导入开仓记录。
2. 导入平仓记录。
3. 自动匹配平仓对应的开仓。
4. 无法匹配时进入待确认状态，不直接污染正式数据。

### 9.2 导出

导出到正式台账模板时，需要支持两种口径：

1. 明细口径：开仓一行，平仓一行，完整保留每次交易。
2. 汇总口径：按业务编码、日期、方向等规则合并，用于对齐当前线下 Excel 习惯。

当前已确认的上海钧能已结算持仓逻辑是：开仓和平仓分别输出独立行，平仓行使用与对应开仓完全一致的业务编码。

## 10. 数据迁移要求

对现有数据迁移时，应遵循以下原则：

1. 当前未平仓记录迁移为开仓持仓，剩余数量等于原始开仓数量。
2. 当前已结算记录迁移为一条开仓持仓加一条平仓明细。
3. 如果旧数据缺少平仓数量，默认按全平处理。
4. 迁移前需要备份原表。
5. 迁移后需要校验总开仓数量、已平数量、剩余数量和已结算损益是否一致。

## 11. 权限和日志

1. 新增开仓、编辑开仓、录入平仓、撤销平仓、删除记录都应写入操作日志。
2. 删除或撤销已结算记录应保留审计痕迹。
3. 普通用户和管理员权限沿用现有系统规则。
4. 对影响损益的字段修改，需要记录修改前和修改后的值。

## 12. 验收标准

本次重构完成后，至少需要通过以下验收：

1. 原有上海钧能页面可正常打开。
2. 可以新增一笔开仓，并在今日交易和即期持仓中看到。
3. 可以对该持仓完整平仓，并在今日交易和本月已结算中看到平仓记录。
4. 完整平仓后，即期持仓不再显示该笔记录。
5. 可以对一笔持仓做两次部分平仓。
6. 部分平仓后，即期持仓显示正确剩余数量。
7. 两次平仓都能在已结算持仓中逐笔显示。
8. 每笔平仓损益计算方向正确。
9. 开仓手续费分摊正确。
10. 导出的台账模板能形成开仓行和平仓行。
11. 现有历史数据迁移后，汇总结果与迁移前一致。
12. 浏览器页面无明显遮挡、错位或无法操作的问题。

## 13. 建议实施阶段

### 第一阶段：数据结构重构

1. 新增开仓持仓表和平仓明细表。
2. 保留旧接口兼容层，避免一次性破坏现有页面。
3. 编写数据迁移脚本。
4. 增加基础单元测试。

### 第二阶段：后端接口重构

1. 新增开仓接口。
2. 新增平仓接口，支持平仓数量。
3. 新增持仓明细接口。
4. 调整今日交易、即期持仓、已结算持仓查询逻辑。

### 第三阶段：页面重构

1. 重做智能台账主页面布局。
2. 优化新增开仓和平仓弹窗。
3. 增加持仓明细抽屉或弹窗。
4. 增加部分平仓显示能力。

### 第四阶段：导入导出打通

1. 对接正式台账模板导出逻辑。
2. 验证业务编码生成规则。
3. 验证已结算持仓开平分行输出。
4. 增加真实样例 Excel 回归测试。

### 第五阶段：上线验证

1. 使用测试数据验证一开一平。
2. 使用测试数据验证一开多平。
3. 使用历史数据迁移样本验证汇总口径。
4. 在测试环境确认无误后，再考虑生产发布。

## 14. 暂不纳入本次范围

以下内容暂不作为本次第一版强制范围：

1. 期权台账逻辑重构。
2. 自动识别线下 Excel 中复杂分批平仓关系。
3. 多账户、多主体统一持仓归并。
4. 自动行情接口深度改造。
5. 业务编码 `-1`、`-2` 批次后缀的强制生成。

## 15. 待确认问题

后续实施前建议确认以下问题：

1. 上海钧能每个品种的合约单位是否已有统一配置表。
2. 部分平仓时开仓手续费是否固定按数量比例分摊。
3. 已结算持仓页面默认展示明细口径，还是同时提供汇总口径。
4. 导出正式模板时是否需要保留分笔平仓后缀。
5. 历史旧数据中是否存在已部分平仓但未完整记录数量的情况。
