# 实时信息汇总历史缓存回填设计

日期：2026-06-23  
状态：设计稿，待实施计划确认  
适用环境：先在 Staging 验证，Production 发布需单独确认

## 背景

`信息预警管理 > 实时信息汇总` 的卡片包含今日值、昨日值、前日值、均值、最小值、最大值、标准差等字段。其中今日值来自实时行情，昨日值和统计字段依赖历史行情缓存。

当前 Web 版已经迁移了桌面版缓存，并保留了 `daily_prices` 和 `calculated_data` 两张表。但现状是：

- `daily_prices` 主要来自旧桌面缓存导入，实时信息汇总没有持续补充最新历史行情。
- `calculated_data` 可以从已有 `daily_prices` 反算，但不能在基础行情缺失时主动拉历史价格。
- 新增 `掉期月差` 后，FE 铁矿石历史行情尚未接入，因此只能显示今日值。
- 上海钧能台账已有自己的价格刷新逻辑，但它服务于持仓和盈亏，不应承担实时信息汇总历史缓存职责。

本设计目标是补齐实时信息汇总自己的历史行情回填能力。

## 目标

为实时信息汇总建立统一的历史缓存回填机制，使所有卡片的历史统计数据可以持续滚动更新。

目标包括：

- 覆盖实时信息汇总模块内全部指标。
- 统一使用现有 `daily_prices` 作为历史基础行情缓存。
- 统一使用现有 `calculated_data` 作为指标计算结果缓存。
- 旧桌面缓存只作为初始化种子，未来不再作为独立数据源并行存在。
- 提供一个轻量入口用于人工刷新和观察状态，不做完整历史明细管理页。
- 补充 FE 铁矿石历史行情能力，为 `掉期月差`、`内外盘差`、`内外盘差2` 提供基础。

## 非目标

第一版不做以下内容：

- 不新增完整的历史行情明细页面。
- 不做 `daily_prices` 明细分页、筛选、导出、人工改价。
- 不把该能力放进上海钧能台账模块。
- 不直接改 Production 数据。
- 不用实时今日值伪造历史收盘价或历史统计。

## 覆盖范围

实时信息汇总下全部指标都纳入同一套历史缓存机制：

- `卷螺差`
- `螺矿比`
- `煤矿比`
- `盘面钢厂利润`
- `月差`
- `掉期月差`
- `内外盘差`
- `内外盘差2`

这些指标都应能通过同一套回填框架维护历史基础价格和计算结果。实现可以分阶段，但设计边界必须覆盖全模块。

## 数据源和缓存收敛

未来唯一的数据源关系定义如下：

- `daily_prices`：实时信息汇总的唯一历史基础行情缓存表。
- `calculated_data`：实时信息汇总的唯一指标计算结果缓存表。
- 旧桌面版 `data_cache.db`：只作为迁移种子导入，不作为长期并行数据源。
- 新回填机制：直接补齐和更新上述两张表。

也就是说，系统不再维护“旧缓存”和“新缓存”两套长期来源。旧缓存导入后即成为 `daily_prices` / `calculated_data` 的一部分，后续由 Web 回填机制持续接管。

## 冲突处理规则

为避免新回填破坏旧缓存，写入规则必须幂等且保守。

### `daily_prices`

唯一键：

```text
info_type + contract_code + calc_date
```

写入规则：

- 新数据有效时，使用 upsert 写入或更新同 key 价格。
- 新数据为空、缺失、无法解析时，不用空值覆盖旧价格。
- 旧缓存有数据而新源暂时没有数据时，保留旧缓存。
- 不同模块通过 `info_type` 隔离，例如 `sh_junneng` 不应与 `卷螺差`、`月差` 混用。

### `calculated_data`

唯一键：

```text
info_type + year + month + calc_date
```

写入规则：

- `calculated_data` 是从 `daily_prices` 推导出的结果，可以在回填后重新计算并覆盖同 key 旧结果。
- 如果基础价格不足，不写入空计算结果覆盖旧结果。
- 计算窗口按滚动交易日窗口处理，目标为最近约 180 个交易日。

## 模块职责

该能力归属：

```text
信息预警管理 > 实时信息汇总
```

不归属：

```text
台账管理 > 上海钧能台账
```

上海钧能台账的价格刷新只服务持仓当前价和盈亏。实时信息汇总回填负责指标统计所需的历史基础行情，两者可以共用 `daily_prices` 表结构，但必须通过 `info_type` 分离业务语义。

## 后端设计

新增独立服务模块，例如：

```text
backend/app/info_summary_backfill.py
```

职责拆分：

- 生成回填任务。
- 按指标识别所需合约。
- 拉取历史行情。
- 写入 `daily_prices`。
- 基于 `daily_prices` 重新计算 `calculated_data`。
- 返回指标级状态。

现有 `calculate_missing_cache_from_prices()` 可继续保留，用于“基础价格已存在时补计算结果”。新增回填服务负责“基础价格也缺失时，先补历史价格”。

## 指标适配器

后端应采用“统一回填框架 + 指标适配器”的结构。

统一框架负责：

- 日期窗口。
- 交易日集合。
- 缓存缺口判断。
- 写入 `daily_prices`。
- 批量计算 `calculated_data`。
- 状态返回和错误隔离。

指标适配器负责：

- 指标需要哪些合约。
- 使用连续合约还是具体合约。
- 使用国内行情源还是 SGX FE 行情源。
- 指标值如何计算。

第一版适配规则：

| 指标 | 历史基础合约 | 说明 |
| --- | --- | --- |
| 卷螺差 | `HC0`, `RB0` | 复刻原 Python 连续合约逻辑 |
| 螺矿比 | `RB0`, `I0` | 复刻原 Python 连续合约逻辑 |
| 煤矿比 | `JM0`, `I0` | 复刻原 Python 连续合约逻辑 |
| 盘面钢厂利润 | `RB0`, `I0`, `J0` | 复刻原 Python 连续合约逻辑 |
| 月差 | `I{yy}{month1}`, `I{yy}{month2}` | 具体合约 |
| 掉期月差 | `FE{yy}{month1}`, `FE{yy}{month2}` | 新增 FE 具体合约 |
| 内外盘差 | `I{yy}{month}`, `FE{yy}{month}`, `USD/CNH` | 多月份，含汇率 |
| 内外盘差2 | `I{yy}{month}`, `FE{yy}{month}`, `USD/CNH` | 多月份，含汇率 |

## FE 历史行情

FE 是新增的数据源难点，不能假设与国内期货使用同一历史接口。

设计要求：

- 抽象出 FE 历史行情获取函数。
- 实现前必须验证可稳定访问的数据源。
- 若 FE 历史源失败，回填结果应明确标记失败或部分失败。
- 失败时不伪造历史数据，不用今日实时价填充历史。
- FE 历史回填成功后，写入 `daily_prices` 的 `contract_code` 使用 `FE2609`、`FE2701` 这类明确合约代码。

## 接口设计

新增状态接口：

```text
GET /api/info-summary/cache/status
```

返回内容：

- `cache_counts`
- 每个指标的 `daily_prices` 最新日期
- 每个指标的 `calculated_data` 最新日期
- 上次回填状态
- 是否存在缺口

新增回填接口：

```text
POST /api/info-summary/cache/backfill
```

第一版参数：

```json
{
  "info_type": null,
  "calc_date": "2026-06-23",
  "force": false
}
```

说明：

- `info_type = null` 表示回填全部实时信息汇总指标。
- `info_type` 有值时只回填单个指标。
- `calc_date` 默认为当天。
- `force = false` 时只补缺口；`true` 时可重算已有结果，但不应以空值覆盖有效历史价。

返回内容：

```json
{
  "status": "success",
  "results": [
    {
      "info_type": "月差",
      "status": "success",
      "price_rows_written": 120,
      "calculated_rows_written": 60,
      "latest_price_date": "2026-06-22",
      "latest_calculated_date": "2026-06-23",
      "message": "已更新"
    }
  ]
}
```

状态值：

- `success`
- `partial`
- `failed`
- `skipped`

## 前端设计

不新增完整页面，只在实时信息汇总顶部增加轻量入口。

建议元素：

- `刷新历史缓存` 按钮。
- `历史缓存截至：yyyy-mm-dd`。
- `上次回填：成功 / 部分失败 / 失败`。
- 保留已有缓存计数展示。

卡片状态可更明确：

- `已读取缓存并刷新今日值`
- `历史缓存已回填`
- `历史价格缺失，仅显示今日值`
- `FE 历史源失败，仅显示今日值`

用户不需要看到每条历史价格明细。

## 自动触发规则

第一版建议以手动回填为主：

- 页面加载时不自动执行大规模联网回填。
- 点击 `刷新历史缓存` 时执行全模块回填。
- 单卡片计算时继续优先读取缓存；缓存缺失时可尝试从已有 `daily_prices` 计算，但不自动触发大规模联网回填。
- 稳定后再考虑接入后台定时任务，例如每日收盘后自动回填。

这样便于 Staging 验证，也避免用户打开页面时触发不可控的慢任务。

## 权限和环境

- 只有具备 `info_summary` 编辑权限的用户可以触发回填。
- Staging 开发和验证只使用 Render `ltm-web-staging` 和 Supabase `LTM WEB STAGING`。
- Production 发布必须单独确认。
- 回填会写业务缓存数据，但不涉及表结构时不需要数据库迁移。
- 若新增回填运行记录表，则属于数据库结构变更，需单独记录并验证。

## 回填运行记录

建议新增轻量运行记录，用于排查，不作为行情数据源。

可选表：

```text
info_summary_backfill_runs
```

字段建议：

- `id`
- `started_at`
- `finished_at`
- `status`
- `triggered_by`
- `scope`
- `message`
- `result_json`

该表只记录回填任务状态，不参与指标计算。

## 验收标准

Staging 第一版验收：

- 点击 `刷新历史缓存` 后，全部指标返回清晰状态。
- `daily_prices` 最新日期能推进到最近可用交易日。
- `calculated_data` 最新日期能按当前计算日更新。
- 旧缓存存在时，新回填不会用空值覆盖有效旧数据。
- 月差能基于具体 I 合约显示昨日、前日和统计值。
- 掉期月差在 FE 历史源可用时能显示历史统计；不可用时明确提示。
- 内外盘差和内外盘差2纳入统一状态，不被静默遗漏。
- 原有今日值实时计算不受影响。
- 上海钧能台账价格刷新不受影响。
- 测试版验证完成后更新 `版本更新记录.md`。

## 分阶段实施建议

第一阶段：

- 建立统一回填服务和轻入口。
- 接入国内连续合约和月差具体 I 合约。
- 写入现有 `daily_prices` 和 `calculated_data`。
- 输出全指标状态，其中 FE 相关指标允许返回数据源未就绪。

第二阶段：

- 验证并接入 FE 历史源。
- 补齐 `掉期月差`。
- 补齐 `内外盘差` 和 `内外盘差2` 的 FE 与汇率历史。

第三阶段：

- 稳定后接后台定时任务。
- 优化运行记录和状态展示。

## 风险

- FE 历史源不稳定，可能导致掉期月差和内外盘相关指标阶段性只能显示今日值。
- 国内历史源与旧桌面缓存存在价格差异时，重算后 `calculated_data` 可能变化。
- 回填任务可能较慢，第一版应避免页面加载自动触发。
- 若一次回填全指标，外部行情源可能有频率限制，需要做超时和部分失败处理。

## 待实施前确认

- FE 历史行情源最终采用哪个接口。
- 第一版是否新增 `info_summary_backfill_runs` 表，还是先只返回即时状态。
- 内外盘差和内外盘差2第一阶段是否只纳入状态，还是同步完成完整历史回填。
