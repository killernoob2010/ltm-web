# 铁矿石期现自动同步可靠性修复业务需求说明

## 1. 文档信息

- 文档状态：待审阅草案
- 适用模块：铁矿石期现数据自动同步
- 首次实施环境：Render Staging 服务 `ltm-web-staging`
- 关联原需求：`2026-07-13_iron-ore-basis-spot-futures-business-requirements.md`
- 本文只补充自动同步的容错、状态记录和恢复规则，不修改原需求中的数据范围、计算公式、展示口径和权限规则。

## 2. 业务背景

测试版铁矿石期现自动同步已能够正常登录 EBC、读取 EBC 数据、读取 I0 行情并完成基差计算，但外部 EBC 请求存在偶发 HTTP 失败。

当前自动任务遇到单次 HTTP 失败后，会直接将本批次记录为失败；同一固定时点对应的任务记录一旦存在，后续调度不会再次执行该批次。因此，短暂的外部请求异常可能导致当日 21:30、次日 09:30 或 10:30 的同步连续缺失，需要人工介入才能恢复。

2026-07-15 测试版排查结果表明：同一代码、同一环境、同一目标日期下，首次只读试运行失败，后续再次试运行可成功取得 105 个源数据点并生成 96 条有效组合。这说明当前主要问题是“偶发请求失败后缺少自动恢复能力”，而不是账号失效、请求参数固定错误或计算逻辑整体不可用。

## 3. 修改目标

本次修改需要实现以下业务目标：

1. 外部数据源发生短时网络或服务异常时，系统能够在有限次数内自动重试。
2. 固定时点批次首次失败后，系统能够继续自动补偿，而不是永久跳过该批次。
3. 自动重试不得产生重复源数据、重复期现结果或重复覆盖已确认的正确数据。
4. 运行记录能够明确反映失败阶段、重试次数和最终结果，同时不得记录任何密码、令牌、Cookie、数据库连接串或完整外部响应正文。
5. 测试版最新数据日期、数据库结果和网页展示保持一致，并能够支持判断数据是否至少更新到 T-1 交易日。

## 4. 修改范围

### 4.1 本期包含

- EBC 登录请求和数据请求的有限自动重试；
- I0 行情请求的有限自动重试；
- 失败批次的自动补偿执行；
- 同一批次的执行次数限制；
- 同步运行状态和脱敏诊断信息补充；
- 试运行与正式写入运行状态口径统一；
- 服务重启后的失败批次恢复；
- Staging 环境的真实数据和网页验收。

### 4.2 本期不包含

- 不修改铁矿石品种、港口、参数版本、规则版本或基差计算公式；
- 不修改 EBC 和 I0 的业务取数口径；
- 不新增人工同步按钮或网页写入入口；
- 不改变当前固定同步时点；
- 不引入新的付费外部服务；
- 不操作 `main`、Production、正式数据库或正式环境变量；
- 不执行任何真实交易、下单或资金操作。

## 5. 业务术语

- **目标日期**：本批次计划获取并计算的业务日期。
- **固定时点**：当前系统约定的自动同步执行时点，例如 T 日 21:30、T+1 日 09:30 和 10:30。
- **批次**：由固定时点和目标日期唯一确定的一次同步任务。
- **请求级重试**：单次批次执行过程中，对同一个外部 HTTP 请求进行有限次数重试。
- **批次级补偿**：某批次执行失败后，由后续调度再次执行该批次。
- **源数据点**：从 EBC 或 I0 获得并通过基础解析的数据记录。
- **有效组合**：完成现货、参数和期货匹配后，可写入期现结果表的港口与品种组合。

## 6. 业务规则

### 6.1 固定同步时点

- 保持现有自动同步时间安排不变。
- 每个固定时点仍生成独立批次，不因前一时点失败而取消后续固定时点。
- 后续固定时点成功且最新数据日期已达标时，前一时点的偶发失败可判定为“偶发失败但已恢复”。

### 6.2 请求级重试

- EBC 登录、EBC 数据查询和 I0 行情查询均应支持请求级重试。
- 每个请求默认最多执行 3 次，包括首次请求和最多 2 次重试。
- 建议重试间隔为首次失败后 3 秒、第二次失败后 8 秒；具体间隔可在技术设计阶段微调，但不得形成无限重试。
- 以下异常允许自动重试：连接失败、连接中断、请求超时、HTTP 429、HTTP 5xx。
- 以下异常不进行请求级重试：配置缺失、HTTP 401、HTTP 403、明确的登录拒绝、响应结构不符合业务协议、业务参数校验失败。
- 重试过程中不得把密码、令牌、Cookie、请求头、数据库连接串或完整响应正文写入日志和运行记录。

### 6.3 批次级补偿

- 已成功的批次不得重复执行。
- 已失败的批次允许由后续调度继续执行，不得仅因同一 `slot_key` 已存在就永久跳过。
- 同一批次默认最多执行 3 次，包括首次执行和最多 2 次补偿执行。
- 批次补偿沿用当前调度检查频率执行，不新增独立高频定时任务。
- 批次成功后立即停止补偿；达到最大执行次数后停止自动补偿，并将该批次标记为需要人工排查。
- 服务重启后，如果当前应执行批次仍处于失败状态且未达到最大执行次数，应继续补偿；不得因进程重启清空执行次数。
- 新固定时点批次的执行不受旧批次达到最大次数影响。

### 6.4 数据幂等与覆盖规则

- 请求级重试和批次级补偿均不得产生重复业务记录。
- 期现结果仍按原业务唯一键执行幂等写入。
- 同一批次重试时可以更新该批次的运行状态，但不得重复累计已写入组合数。
- 失败发生在数据库事务提交前时，本次写入必须整体回滚，不保留半批数据。
- 已成功写入的目标日期再次被其他固定时点校验时，应按现有幂等规则处理，不增加重复行。
- 本次修改不得改变历史数据的计算值和原始业务口径。

### 6.5 运行状态

同步批次统一使用以下状态：

- `success`：目标范围内可计算组合全部成功写入，且不存在应写未写的组合。
- `partial`：已生成并写入部分有效组合，但存在明确跳过或缺失的应处理组合。
- `failed`：未形成可用写入结果，或执行在关键阶段终止。
- `skipped`：该批次无需执行，例如已经成功完成；不得用 `skipped` 掩盖失败。

试运行和正式写入必须采用相同的状态判定规则。试运行存在 `combinations_skipped > 0` 时，不得仅因已有计算结果就标记为 `success`。

### 6.6 诊断与审计信息

每个批次至少保留以下可审计信息：

- `slot_key`
- `status`
- `target_start_date`
- `target_end_date`
- `source_points_seen`
- `source_points_inserted`
- `source_differences`
- `combinations_written`
- `combinations_skipped`
- `error_code`
- `attempt_count`
- `error_stage`
- `http_status`
- `last_attempt_at`

诊断字段规则：

- `error_stage` 只记录失败阶段，例如 `ebc_login`、`ebc_query`、`i0_query`、`calculate`、`database_write`；
- `http_status` 只记录 HTTP 状态码，不记录响应正文；
- `error_code` 使用稳定、可汇总的错误编码，不使用包含敏感信息的原始异常文本；
- 成功批次应保留实际执行次数，便于识别“首次成功”和“重试后成功”；
- 日志和数据库中均不得保存或展示账号密码、令牌、Cookie、数据库连接串及其他密钥。

### 6.7 页面一致性与业务判断

- “期现数据展示”页面顶部的“最新数据日期”必须与数据库 `iron_ore_basis_results` 的最大业务日期一致。
- 页面显示组合数时，应与该最新业务日期在数据库中的有效结果行数一致。
- 自动同步是否达标，以最新数据日期是否至少达到 T-1 交易日为主要判断依据。
- 单个固定时点出现 EBC `http_error`，但后续批次成功且最新数据日期达标时，判定为“偶发失败但已恢复”。
- 三个固定时点均失败、最新数据日期落后，或页面与数据库不一致时，判定为“需要人工排查”。
- 页面不得为了显示“最新”而补写不存在的日期或伪造数据更新时间。

## 7. 异常处理要求

| 异常场景 | 系统处理 | 最终业务结果 |
| --- | --- | --- |
| 首次请求超时，第二次成功 | 自动重试并继续本批次 | 按实际结果记录 `success` 或 `partial`，执行次数大于 1 |
| 连续发生连接失败或 HTTP 5xx | 完成请求级重试后将本次批次执行记为失败 | 未达到批次上限时等待后续调度补偿 |
| HTTP 429 | 按有限间隔重试 | 超过请求上限后进入批次级补偿 |
| HTTP 401、403 或明确登录拒绝 | 不进行请求级重复登录轰炸 | 本次记为失败，保留脱敏错误码并提示人工检查配置或授权 |
| 响应结构异常 | 不写入不可信数据 | 本次记为失败并记录失败阶段 |
| 部分组合缺失 | 写入有效组合并记录跳过数 | 状态记为 `partial` |
| 数据库写入失败 | 回滚本批次写入 | 状态记为 `failed`，不得留下半批结果 |
| 同一批次已成功 | 不重复执行 | 状态保持成功，不新增重复记录 |
| 同一批次失败但未达上限 | 后续调度再次执行 | 成功后停止补偿 |
| 同一批次达到最大执行次数 | 停止自动补偿 | 标记“需要人工排查” |

## 8. 验收标准

### 8.1 自动重试

- 模拟首次连接失败、第二次成功时，批次能够自动完成，且运行记录显示实际执行次数。
- 模拟持续连接失败时，请求和批次均在达到上限后停止，不得无限循环。
- 模拟 HTTP 401、403、登录拒绝或响应结构异常时，不执行不适用的请求级重试，不写入错误数据。

### 8.2 批次恢复

- 同一 `slot_key` 首次失败后，后续调度能够再次执行。
- 同一 `slot_key` 成功后，后续调度不再重复执行。
- 服务重启后，未达到最大次数的失败批次能够继续补偿。
- 达到最大执行次数的失败批次停止自动补偿，并能从运行记录中明确识别。

### 8.3 数据正确性

- 重试前后业务唯一键数量不增加，不产生重复结果。
- 数据库写入异常时不保留半批数据。
- `success`、`partial`、`failed` 和 `skipped` 在试运行与正式写入中判定一致。
- 原有基差公式、品种、港口、参数版本和规则版本无回归。

### 8.4 Staging 真实验收

- 使用 Render Staging 的真实配置执行，不读取、展示或记录任何密钥。
- 以 2026-07-14 为目标日期的只读试运行应能够取得约 105 个源数据点并生成 96 条有效组合；如外部源实时数据变化，以运行记录和源数据差异说明为准。
- 经允许的 Staging 自动任务完成后，数据库最新业务日期至少达到对应的 T-1 交易日。
- 最新业务日期的数据库组合数与测试版网页展示一致。
- 测试版页面顶部“最新数据日期”与数据库最大业务日期一致。
- Staging 验收通过后停留在发布门禁，不自动进入 Production。

## 9. 安全与发布边界

- 开发、测试、部署和数据验证首先只在 `staging` 分支及 Render Staging 环境进行。
- 不得在日志、运行记录、需求文档、版本记录或测试报告中写入任何密钥。
- 不得通过人工频繁触发同步来替代自动恢复能力的验收。
- Production 发布、正式环境变量变更和正式数据库操作必须另行评估并获得用户明确确认。
- 本次修改应支持快速回滚到修改前版本；回滚不得删除已有正确期现数据。

## 10. 审阅决策点

请重点确认以下规则：

1. 是否同意请求级最多 3 次，建议间隔 3 秒、8 秒。
2. 是否同意同一批次最多执行 3 次，并沿用当前调度检查频率补偿。
3. 是否同意 HTTP 401、403、明确登录拒绝和响应结构异常不做请求级重试。
4. 是否同意运行记录补充 `attempt_count`、`error_stage`、`http_status`、`last_attempt_at`。
5. 是否同意试运行存在跳过组合时统一标记为 `partial`。
6. 是否同意本次只完成 Staging 修复和验收，Production 另行决策。
