# Web 版迁移执行计划

## 1. 当前阶段交付物

本阶段只完成迁移前置梳理和架构设计，不启动源码改造。

已形成的文档：

- `docs/01_current_system_inventory.md`：现有桌面版功能和迁移边界。
- `docs/02_web_architecture.md`：Web 版目标架构、模块拆分、API 草案、运行方式。
- `docs/03_data_model_draft.md`：Web 版数据结构草案。

## 2. 建议开发阶段

### 阶段 1：项目骨架和权限闭环

目标：

- 建立后端 FastAPI 项目。
- 建立前端 React/Vue 项目。
- 实现登录、退出、当前用户、菜单权限。
- 实现用户管理、权限管理、操作日志。

验收：

- 管理员可登录并看到全部菜单。
- 普通用户只能看到有查看权限的菜单。
- 无编辑权限时不能调用写接口。
- 登录、用户变更、权限变更写入操作日志。

### 阶段 2：上海钧能台账

目标：

- 迁移 `trades` 和 `daily_prices`。
- 实现交易列表、筛选、新增、编辑、删除、平仓、价格刷新、导出。
- 已移植功能涉及离岸人民币汇率时，优先使用 SGX USD/CNH 对应月份合约价格；旧的新浪离岸汇率加月份调整逻辑仅作为备用。后续继续迁移未完成模块时，汇率接入也应沿用这一规则。
- 抽离盈亏和资金收益计算。

验收：

- 使用同一组样例数据，Web 版盈亏结果与桌面版一致。
- 未平仓和已平仓状态正确。
- 无编辑权限用户无法新增、编辑、删除和平仓。

### 阶段 3：钢材出口套保和子公司套保

目标：

- 迁移开仓表、平仓表和操作日志。
- 实现挂单、确认开仓、持仓、确认平仓、已平仓、汇总、导出。
- 抽离共同的套保台账服务逻辑。

验收：

- 部分平仓后剩余数量正确。
- 全部平仓后状态正确。
- 美元和人民币收益计算与桌面版一致。
- 订单号生成规则稳定且不会重复。

### 阶段 4：期权交易

目标：

- 迁移 `option_trades`。
- 复用 Greeks 计算和到期日规则。
- 实现开仓、编辑、删除、平仓、行情刷新、导出。

验收：

- Greeks 计算与桌面版一致。
- 部分平仓和全部平仓行为与桌面版一致。
- 行情获取失败时页面给出明确错误，不影响已有数据查看。

### 阶段 5：信息汇总和风险预警

目标：

- 迁移指标计算、行情缓存、交易日、预警规则、预警历史。
- 将桌面线程改为后端定时任务。
- 实现预警通知和历史查看。

验收：

- 手动计算指标结果与桌面版一致。
- 定时扫描能写入预警历史。
- 停用规则不触发预警。
- 页面能展示最新值和历史记录。

### 阶段 6：后台数据管理

目标：

- 实现备份、查看、下载、恢复、自动备份配置。
- 实现导入导出任务记录。
- 恢复前自动创建紧急备份。

验收：

- 只有管理员可操作备份恢复。
- 备份文件可下载。
- 恢复动作有二次确认和操作日志。
- 恢复失败不会破坏当前数据。

## 3. 核心测试场景

权限：

- 管理员拥有全部权限。
- 普通用户无查看权限时菜单不可见。
- 普通用户有查看无编辑时只能查看。
- 直接调用无权限接口返回 403。

台账：

- 新增开仓。
- 编辑开仓。
- 删除记录。
- 部分平仓。
- 全部平仓。
- 查询和筛选。
- 汇总行计算。
- 导出 Excel。

行情和预警：

- 手动刷新行情。
- 行情源失败。
- 自动任务刷新。
- 预警触发。
- 预警停用。
- 预警历史已读。

数据管理：

- 创建备份。
- 查看备份内容。
- 下载备份。
- 恢复备份。
- 恢复前紧急备份。
- 导入文件格式错误。

## 4. 关键风险

- 当前桌面代码中业务逻辑和 Tkinter 界面混杂，不能机械搬运。
- 多个模块都有各自 SQLite 连接和建表逻辑，需要统一数据访问层。
- 钢材出口套保和子公司套保高度相似，但不能过早合并业务表。
- 信息预警依赖线程、实时行情和缓存，Web 化后需要重新设计任务生命周期。
- 默认管理员账号和 SHA-256 密码哈希不适合作为正式 Web 安全方案。
- 备份恢复在 Web 环境风险更高，必须加权限、确认、日志和紧急备份。

## 5. 下一步建议

先把现有源码解压到工作目录或建立独立代码仓库，然后进入阶段 1。阶段 1 的实现不要同时改所有业务模块，只先完成系统骨架、登录权限和菜单，这样后续每个模块都能按同一套权限和页面框架接入。
